# Risk Overlay 設計（MarketRegime・EventRisk）

本ドキュメントは Issue #132 の設計方針を、株の専門用語の解説付きでまとめたものです。

---

## この設計が解決する問題

現在のパイプラインでは、VIX やイベント日程などの市場全体の情報は「コメント（テキスト）」としてしか使われておらず、**実際のポジションサイズや売買判断の数値には一切反映されていない**。

しかし「計算式に直接組み込む」のは雑すぎるため、**Risk Overlay（リスクオーバーレイ）** という別レイヤーで制御する方針を採用する。

---

## 用語集

| 用語 | 説明 |
|------|------|
| **ポジションサイズ** | 1つの銘柄に対して「いくら分の株を買うか」の金額。大きいほど積極的、小さいほど慎重 |
| **VIX（恐怖指数）** | S&P500（米国の代表的な株価指数）の今後30日間の値動きの激しさを予測する指標。高いほど市場が不安定。ただし個別銘柄の値動きとは直接連動しない |
| **RSP/SPY比率（breadth）** | SPY＝大型株の影響が大きい指数、RSP＝全銘柄を均等に見た指数。この比率で「上昇が幅広い銘柄に及んでいるか、一部の大型株だけか」がわかる |
| **breadth（ブレス）** | 「市場の広がり」。上昇している銘柄が多ければ breadth が良い、少なければ悪い |
| **Implied Move（期待変動幅）** | オプション市場（株を将来の価格で売買する権利の市場）が織り込んでいる、今後の値動き幅の予測。決算発表前などに大きくなる |
| **ATR（Average True Range）** | 過去の値動き幅の平均。「この銘柄は普段どれくらい動くか」を数値化したもの |
| **エクスポージャ** | リスクにさらされている金額の総量。全銘柄のポジションサイズの合計 |
| **bps（ベーシスポイント）** | 0.01%のこと。35bps = 0.35% |
| **β（ベータ）** | 市場全体が1%動いたとき、その銘柄が何%動くかの指標。β=1.5 なら市場の1.5倍動く |
| **RR比（リスクリワード比）** | 損切り幅に対して利益確定幅がどれだけあるか。RR比2.0 なら「損1に対して利益2を狙う」 |
| **損切り（ストップロス）** | 株価が想定と逆に動いたとき、損失を限定するために売ること。事前に「ここまで下がったら売る」と決めておく |
| **scale-in（分割エントリー）** | 一度に全額買わず、複数回に分けて買うこと。リスクを分散できる |
| **confidence** | Analyzer の議論で「買い」「売り」のどちらがどれだけ支持されたかの度合い。投票差が大きいほど高い |
| **thesis（テーゼ）** | 「この銘柄を買うべき（売るべき）理由」の仮説。例：「AIブームでNVDAの売上は伸び続ける」 |
| **Shadow Mode** | 実際の売買には反映せず、「もし適用したらどうなっていたか」を記録だけする試運転モード |
| **Scenario Planner** | 1つのプランだけでなく、状況別に複数のプランを事前に用意しておく手法 |
| **gap down / gap up** | 株価が前日の終値から大きく離れた位置で翌日の取引が始まること。gap down＝大幅に下で始まる、gap up＝大幅に上で始まる |
| **Offline Calibration** | 過去の実績データを使って、閾値や係数を事後的に調整・改善すること |
| **implied correlation** | オプション市場から逆算した、銘柄間の値動きの連動度。高いと「みんな一緒に動く」状態で分散効果が薄れる |
| **dispersion（ディスパージョン）** | 銘柄間の値動きのばらつき。高いと個別銘柄がバラバラに動いている状態 |

---

## 核心：2つのレイヤーを分ける

```
Analyzer confidence：「この銘柄の分析はどれだけ確かか？」
Risk overlay：      「今の市場環境で、どこまで賭けていいか？」
```

この2つを**絶対に混ぜない**。

```
base_position  = f(confidence, budget, risk_limit)   ← 分析の質で決まる
final_position = base_position × regime_cap × event_cap  ← 環境で絞る
```

こうすると、あとから振り返ったとき：
- 「分析は当たっていたが、環境制約で小さく買ったから利益が少なかった」
- 「分析は外れたが、環境制約でサイズを絞っていたから損失は軽かった」

のように**原因を分解して検証できる**。

## MarketRegime：ポートフォリオ全体の制約

### なぜ個別銘柄に直接かけないのか

VIX は「市場全体の不安度」であって、個別銘柄のリスクではない。

- 個別株の変動が激しくても、銘柄間の動きがバラバラなら VIX は低く見える（分散効果）
- 決算が近い銘柄は指数より個別の方がはるかに激しく動く
- 同じ VIX 28 でも「暴落直後の回復局面」と「じわじわ崩れている局面」では意味が全然違う

→ だから VIX を個別銘柄の係数に直接かけるのではなく、**ポートフォリオ全体の上限**として使う。

### 状態の判定（State Machine）

VIX の生の数値ではなく、複数の指標を組み合わせて「状態」に変換する：

| 状態 | どんなとき | イメージ |
|------|-----------|---------|
| **CALM** | VIX 低位安定、breadth 良好 | 穏やかな相場。通常通り投資できる |
| **NORMAL** | 特に異常なし | 普段の相場 |
| **STRESS** | VIX 上昇 + breadth 悪化 | 不穏な空気。慎重にすべき |
| **CRISIS** | VIX 急騰 + 複数条件同時悪化 | パニック状態。新規投資は基本やめる |

### 状態ごとの制約

| 状態 | ポートフォリオ上限 | 新規建て数 | 1銘柄あたりリスク上限 | 備考 |
|------|------------------|-----------|---------------------|------|
| CALM | 制限なし | 制限なし | 50bps | |
| NORMAL | 制限なし | 制限なし | 50bps | |
| STRESS | 総額の60%まで | 1銘柄まで | 35bps | 分割エントリー強制 |
| CRISIS | 総額の25%まで | 新規停止 | 15bps（例外時のみ） | 高ボラ時はストップ注文が想定通り約定しない可能性があるため保守的に設定 |

### MarketRegime が制約すべきもの / すべきでないもの

**制約先として適切：**
- 総エクスポージャ上限
- 新規建て数の上限
- 1銘柄あたり最大リスク額
- セクター集中上限（同じ業種に偏りすぎない）
- 分割エントリー強制
- 現金比率の下限（最低限キャッシュを残す）

**制約先として不適切：**
- 個別銘柄の期待リターン補正
- 銘柄ごとの confidence そのもの
- 「VIX 高いからこの銘柄は買いじゃない」といった直接判定

### 重要な注意：複数条件が揃ったときだけ強く締める

VIX 単体で判断しない。VIX が高くても breadth が改善中なら回復局面かもしれない。**複数の指標が同時に悪化したときだけ STRESS / CRISIS に移行する**。単一指標の閾値超えだけで強い制約を発動させると、誤発動が増える。

### 制約の出力イメージ

```yaml
regime_state: STRESS
portfolio_gross_cap: 0.60    # エクスポージャ上限（60%）
max_new_positions: 1         # 新規は1銘柄まで
max_risk_per_position_bps: 35  # 1銘柄あたりリスク上限 0.35%
force_scale_in: true         # 分割エントリー強制
allow_breakout_entry: false  # 急騰追いかけ禁止
```

## EventRisk：個別トレードの実行ルール

EventRisk は MarketRegime と違い、**個別のトレードの「やるか・やらないか・どのサイズでやるか」** を決めるのに向いている。

### EventRisk の制約出力

| 出力項目 | 説明 |
|----------|------|
| `allow_new_entry` | 新規エントリーを許可するか |
| `allow_hold_through_event` | イベントをまたいで保有し続けてよいか（決算前に持っている株をそのまま持つか） |
| `event_cap` | サイズ縮小係数（0.0〜1.0） |
| `entry_freeze_window` | エントリー凍結期間（イベントの何日前から新規を止めるか） |
| `post_event_cooldown` | イベント後の冷却期間（イベント直後は値動きが荒いので、落ち着くまで待つ期間） |
| `required_stop_method` | ストップ方式の強制（通常の損切りではなく、イベント用の広めの損切り幅を強制するなど） |

### 決算（個別銘柄のイベント）

決算発表前後は個別銘柄の値動きが極端になるため、特別なルールが必要：

| 条件 | 対応 |
|------|------|
| 決算まで ≤ 3営業日（短期トレードの場合） | 新規エントリー停止 |
| 決算まで 4〜7営業日 | ポジションサイズ × 0.7 |
| Implied Move > 損切り幅 | 大幅縮小 or イベント後に回す |

※ 7営業日前でいきなり半減は強すぎるため、4〜7営業日は 0.7倍に緩和。直前3営業日のみ新規停止で十分にリスクを抑えられる。

### マクロイベント（FOMC、CPI、雇用統計等）

これらは個別銘柄ではなく市場全体に影響するため、ポートフォリオ側の制約として扱う。ただし全てのイベントを同じ重さで扱わず、Tier で分ける：

| Tier | イベント | 当日の対応 |
|------|---------|-----------|
| **Tier1**（市場を大きく動かす） | FOMC、CPI、雇用統計 | ポートフォリオ上限 × 0.7 |
| **Tier2**（影響は限定的） | GDP、小売売上高、ISM | ポートフォリオ上限 × 0.85 |

高β銘柄（市場の1.5倍以上動く銘柄）には Tier に関わらず追加で縮小。

### event_pressure（イベント圧力）

「市場が予想する値動き幅」が「自分の損切り幅」を超えるかどうかで判断する：

```
event_pressure = Implied Move ÷ 損切り幅
```

| event_pressure | 意味 | 対応 |
|---------------|------|------|
| < 0.7 | イベントの影響は損切り幅に余裕あり | × 1.0（通常通り） |
| 0.7 〜 1.0 | 損切り幅に近づいてきた | × 0.75 |
| 1.0 〜 1.3 | 損切り幅を超えている | × 0.5 |
| > 1.3 | 損切りが吹き飛ぶ可能性が高い | 新規停止 or イベント後に回す |

※ 3段階だと境界をまたいだときに挙動が急に変わる。4段階にすることで「損切り幅に近づいたら少しずつ絞り、超えたらかなり削る」という段階的な制御ができる。高ボラ時はストップ注文が想定通り約定する保証がないため、早めに絞り始める方が安全。

## 実装アーキテクチャ：Policy Engine 方式

「計算式に VIX を直接入れる」のではなく、**制約を先に決めてから計算する**方式：

```
① Risk Policy Engine（新モジュール）
   MarketRegime + EventRisk → 制約セット（risk_policy）を出力
        ↓
② plan_calc.py（既存）
   confidence で base_size を決め、risk_policy の制約で絞る
        ↓
③ plan-generator（既存）
   発動した制約を commentary で説明する
```

### 利点

- **ルールが見える**：なぜ縮小したか、なぜ停止したかが明確
- **バックテストしやすい**：過去データで「overlay ありなし」の比較が簡単
- **LLM の気分に左右されない**：決定論的なルールで制御
- **commentary と数値が一致する**：制約を説明するので矛盾しない

## 代替アプローチの比較

Policy Engine 以外にも検討された手法がある。将来的に組み合わせることも可能。

### ① State Machine（状態遷移）方式 ← 本設計で採用

MarketRegime と EventRisk をそれぞれ状態に変換し、組み合わせごとにルールを固定する。

```
MarketRegime: CALM / NORMAL / STRESS / CRISIS
EventRisk:    NO_EVENT / PRE_EVENT / EVENT_DAY / POST_EVENT
```

- 長所：実装が簡単、説明しやすい、壊れにくい
- 短所：状態の境界が雑になりやすい（VIX 24.9 と 25.1 で対応が変わる等）

### ② Scenario Planner（シナリオ計画）方式

1つのプランではなく、状況別に複数のプランを事前に用意しておく。

例：
- **今は watch only**（イベント前なので新規エントリーしない）
- **決算後 gap down でも thesis 維持なら** → 再評価して入る
- **gap up で expected move 超過なら** → 追わない（高値掴みのリスク）

- 長所：バイナリーイベント（決算のように「良いか悪いか」で大きく動くイベント）に強い。「今は入らない」を自然に扱える
- 短所：実装がやや重い。プランの分岐管理が必要

### ③ Offline Calibration / Learned Policy（学習型）方式

過去の実績データで「overlay をかけた場合 / かけなかった場合」の成績を比較し、閾値や係数を定期的に更新する。

- 長所：だんだん賢くなる。改善点を Issue 化しやすい
- 短所：十分なデータが必要。過去のパターンに過剰最適化するリスクがある

### 現実的な導入順序

Phase 1 で ① State Machine を導入し、Shadow Mode で運用。データが溜まったら ③ Offline Calibration で閾値を調整。② Scenario Planner は決算シーズンなど特定の局面で必要に応じて追加。

## ログ要件

毎回、以下を分けて記録する：

| 項目 | 内容 |
|------|------|
| `base_size` | overlay 適用前のポジションサイズ |
| `regime_cap` | MarketRegime による縮小係数 |
| `event_cap` | EventRisk による縮小係数 |
| `final_size` | 最終ポジションサイズ |
| `blocked_reason` | エントリー拒否時の理由 |

これがないと「分析が悪かったのか、リスク制御が強すぎたのか」が事後検証できない。

---

## 導入ステップ

### Phase 1：Shadow Mode（まずここから）

本番のポジションサイズは変えず、「もし overlay を適用していたらどうなっていたか」を記録だけする。数週間観察して閾値を調整する。

### Phase 2：本番適用

Shadow Mode の結果を検証してから本番に切り替える。

## Override（例外ルール）

Analyzer が「中長期保有前提」「イベント跨ぎを織り込み済み」と明示している場合は、新規停止ではなく縮小に落とす。機械的に止めるだけでなく、分析の文脈も考慮する。
