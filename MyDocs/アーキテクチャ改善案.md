# アーキテクチャ改善案

DB操作マップの調査（2026-04）で見えた、複雑さを解消できそうな箇所のまとめ。

---

## 1. archive.status の上書き問題（済）

各ブロック（Monitor, Analyzer, Planning）が同じ `archive.status` を "completed" で上書きしている。
「どのブロックが完了したのか」は判別できない。

実際のフロー制御は `active` フラグ + 専用 fetch 関数（`fetch_active_for_analyzer` 等）が担っており、status はログ程度の意味しかない。

ただし `status = "completed"` は `get_latest_archivelog_with_newplan` 等のクエリフィルタに使われているため、単純に削除はできない。

**対応案**：
- 最終ブロック（Planning）だけが "completed" を書く形にする
- または `updated_by` カラムを追加して書き込み元を記録する

---

## 2. mode 判定の重複（済）

holdings.shares を見て mode（"buy" / "review"）を判定するロジックが3箇所に分散している：

| 箇所 | タイミング |
|------|-----------|
| Technical/src/main.py | archive 新規作成時 |
| analyzer_batch.py | Analyzer 起動時 |
| Monitor/src/main.py | 監視チェック時 |

Technical が `archive.mode` に書き込んでおり、後続ブロックはこの値を参照すれば再判定不要。

**対応**：analyzer_batch.py を `archive.mode` 参照に変更済み（af2248e）。

Monitor の holdings チェックは残している。理由：Monitor を `python Monitor/src/main.py --ticker NVDA` で直接実行した場合、Technical が作った archive が存在しないケースがある。その場合 Monitor が自分で新規 archive を作るため、mode を holdings から判定する必要がある。通常の定期実行（main_pipeline 経由）ではこのパスは通らない。

---

## 3. verdict の上書き

Analyzer が `archive.verdict` に判定を書き込み、Planning が同じカラムを上書きする。
Analyzer の元の判定が失われるため、「Analyzer は BUY と言ったのに Planning が止めた」ケース（Issue #150）の判別が難しい。

**対応案**：カラムを分離
- `archive.analyzer_verdict` — Analyzer の判定（BUY, SELL, HOLD 等）
- `archive.planning_verdict` — Planning の最終判定（数値制約を加味した結果）

---

## 4. holdings 書き込みの分散（済）

holdings テーブルへの書き込みが複数経路から行われている：

| 経路 | 書き込み内容 |
|------|-------------|
| Watch → `_sync_holdings_from_logs` | shares, avg_cost（action_log から再計算） |
| ActionLog/data_service.py → `_sync_holdings_from_logs` | 同上（UI 編集時） |
| main_pipeline.py → `upsert_holding` | current_price（Technical の latest_price） |

shares / avg_cost は `_sync_holdings_from_logs` に集約されているが、current_price だけが main_pipeline から別経路で更新されている。

**対応案**：current_price の更新も `_sync_holdings_from_logs` または専用関数に統合し、書き込み元を明確にする

---

## 5. populate_from_archive / populate_from_monitor の重複（済）

ActionLog/src/auto_populate.py に2つの類似関数がある：

| 関数 | 用途 | 呼び出し元 |
|------|------|-----------|
| `populate_from_archive` | newplan_full 付き archive → action_log | Watch |
| `populate_from_monitor` | monitor 結果のみ → action_log | main_pipeline（保有中+OK） |

### 共通部分（まったく同じロジック）
- 二重投入チェック（`list_action_log_archive_ids`）
- holdings 取得（`get_holding`）
- action_log 全件取得 → 株数計算（`calc_total_shares`）
- 前回 cumulative_invested の算出（`get_latest_action_log`）
- total_assets / pnl 計算
- INSERT（`create_action_log`）

### 違う部分（入力ソースの差のみ）
- **populate_from_archive**：newplan_full の YAML をパースして decision/quantity/price/money_in/action_text を抽出
- **populate_from_monitor**：monitor_data から直接取得（decision=結果文字列, quantity=0, money_in=0, price=current_price, action_text=固定テキスト）

**対応案**：共通の累積計算・INSERT ロジックを内部関数 `_insert_action_log_with_calc(ticker, archive_id, row, market, usd_jpy)` に抽出し、2関数は「行の組み立て」だけを担当する形にする

---

## 6. Watch の責務過多（済）

Watch が担っている責務：

1. watchlist 更新
2. Discord 通知
3. ActionLog 投入（populate_from_archive）
4. holdings 同期（_sync_holdings_from_logs）
5. archive.active = false

パイプライン最後の「後片付け全部」になっている。

**対応案**：現状は動いているため優先度は低いが、将来的に ActionLog 投入 + holdings 同期を Watch から切り出し、独立した後処理ステップにすることで責務を明確化できる

---

## 7. 保有状況スナップショットを archive に含める（済）

現状、Monitor・Analyzer・Planning が毎回 holdings テーブルを直接読みに行っている。
パイプライン総則の「ブロック間のデータは archive テーブルを介して受け渡す」に照らすと、参照データも archive に含めた方が一貫性がある。

パイプライン実行中に holdings が変わることはないので、スナップショットで十分。

**対応**：Technical が `archive.holdings_snapshot`（shares, avg_cost）を書き込み、後続ブロックは archive から読む

---

## 優先度

| 優先度 | 項目 | 理由 |
|--------|------|------|
| 高 | 2. mode 判定の統一 | 不整合リスクがある（holdings 変更のタイミングで判定がズレる可能性） |
| 高 | 3. verdict の分離 | Issue #150 の前提として必要 |
| 高 | 7. holdings スナップショット | 総則の伝言板方式を徹底。mode 統一（項目2）と同時に対応できる |
| 中 | 5. populate 関数の統合 | 保守性向上 |
| 低 | 1. status 上書き問題 | 実害が少ない |
| 低 | 4. holdings 書き込み統合 | current_price だけの問題 |
| 低 | 6. Watch 責務分離 | 動いているため急ぎではない |
