# 指標フィルター（IndicatorFilter）設計

## 背景と動機

パイプラインは1日2〜4回、Technical → Monitor → Analyzer → Planning → Watch → ActionLog を全て通している。このうち Monitor は LLM + WebSearch を使う重いブロックで、コストと時間がかかる。テクニカル指標や重要指標が安定している間は Monitor 以降をスキップしたい。

## 設計プロセス

ChatGPT と3回のやり取りを行い、フィルター仕様を策定。

### 主な論点と結論

| 論点 | 結論 | 理由 |
|------|------|------|
| 変化検出 vs 絶対値 | 変化検出を主軸 + 絶対値ハードトリガーを残す | VIX>=30 のような高ストレス状態は水準自体に意味がある |
| スコアリング vs OR 条件 | OR 条件（スコアリングなし） | 重みの根拠がない。調整対象が増えてデバッグしにくい |
| RSI/出来高の扱い | 単独トリガー禁止。組み合わせのみ | RSI は強いトレンドで高止まりする。出来高だけでは好悪不明 |
| 定期フル実行 | 安全ネットとして採用 | フィルターの取りこぼしを回収する仕組みが必要 |
| SMA200 | mid/long のみ新規割れを検出 | ずっと下にいるなら定期フル実行で拾えばよい |

### ChatGPT レビューで修正した点

| 指摘 | 対応 |
|------|------|
| indicator_filter.py に責務を寄せすぎ | loader 層（DB読み取り）と evaluator 層（判定ロジック）に分離 |
| infer_schedule_label() は壊れやすい | 廃止。main_pipeline.py が ScheduleContext を構築して filter に渡す |
| fail-open 仕様が曖昧 | current 欠損 / prev 欠損 / plan 欠損の3パターンを明文化 |
| RSI > 70 AND close < SMA20 は不自然 | v1 から削除 |
| volume 条件の「下落方向」定義が弱い | close < prev_close を追加 |
| COMPLETE と filter skip を区別すべき | FILTER_SKIPPED ステータスを追加 |

## 新しいパイプラインフロー

```
Technical → ImportantIndicators → IndicatorFilter → [トリガー銘柄のみ] Monitor → Analyzer → Planning → Watch → ActionLog
```

- Analyzer 側の `_ensure_important_indicators()` は「取得済みならスキップ」の防御コードがあるため編集不要

## フィルター仕様

### 定期フル実行（安全ネット）

| span | フル実行タイミング |
|------|------------------|
| short | 毎営業日の引け後（JP_PM / US_PM） |
| mid | 月水金の引け後（JP_PM / US_PM） |
| long | 週末（WEEKEND） |

定期フル実行日はフィルターをバイパスし、必ず Monitor に進む。

### 市場全体ゲート（1つでも該当 → 全銘柄 Monitor）

| 条件 | 閾値 |
|------|------|
| VIX 絶対値 | >= 30 |
| VIX 前回差 | +4pt 以上 |
| 米10年債利回り前回差 | ±15bp 以上 |
| 政策金利（FFR / BOJ） | 前回から変化 |
| FOMC / 日銀会合 | 1営業日以内 |

### 個別ハードトリガー（1つでも該当 → その銘柄 Monitor）

| 条件 | 閾値 |
|------|------|
| 株価乖離 | price_tolerance_pct 超え（short:3%, mid:5%, long:7%） |
| イベント近接 | short:5日, mid:3日, long:2日以内 |
| 新決算データ | earnings.latest_quarter が前回から変化 |
| EPS サプライズ | <= -10% |
| 新規 SMA200 割れ | mid/long のみ。前回 close > SMA200 → 今回 close < SMA200 |

### 個別ソフトトリガー（組み合わせで該当 → その銘柄 Monitor）

| 条件 |
|------|
| RSI < 30 かつ close < SMA20 |
| volume_ratio_5d >= 2.0 かつ close < SMA20 かつ close < prev_close |

### 用語定義

- **plan_price**：PlanSpec の `data_checks.current_price`（= プラン作成時点の株価）。price_tolerance_pct はこの値を基準に乖離を測る。anchor_price（基準価格）とは別物。
- **days_to_event の営業日計算**：v1 ではカレンダー日数（暦日）で判定する。JP/US 営業日カレンダー判定は v1.1 で検討。
- **新決算データ判定**：v1 では `earnings.latest_quarter` の変化で判定する。v1.1 で `eps_actual` / `revenue_actual` の変化も判定基準に追加を検討。

### fail-open ルール

| 状況 | 動作 |
|------|------|
| current データ欠損 | Monitor 行き（判定材料なし = 安全側に倒す） |
| prev データのみ欠損（初回実行等） | デルタ判定を無効化、絶対値判定のみ |
| plan 価格欠損 | Monitor 行き（価格チェック不能） |

### v1 で使わないもの（定期フル実行日に回す）

スコアリング、相対強度（3ヶ月）、セクターETF比、Implied Move、ADX、NATR、MACD 単独

## 設計：indicator_filter.py の構造

### loader 層（DB 読み取り）

```python
@dataclass
class TickerInputs:
    ticker: str
    span: str                          # "short" / "mid" / "long"
    current_tech: dict | None          # 今回の archive.technical
    previous_tech: dict | None         # 前回の archive.technical
    current_ii: dict | None            # 今回の archive.important_indicators
    previous_ii_earnings: dict | None  # 前回の earnings（latest_quarter 比較用）
    plan_price: float | None           # latest newplan_full の plan 価格

@dataclass
class FilterInputs:
    current_market: dict | None        # VIX, us_10y_yield, ffr, boj_rate
    previous_market: dict | None       # 前回の市場データ
    central_bank_within_n_days: bool   # FOMC/BOJ が N 日以内か
    tickers: dict[str, TickerInputs]

def load_filter_inputs(tickers, pipeline_start) -> FilterInputs
```

### evaluator 層（純ロジック、DB アクセスなし）

```python
@dataclass
class ScheduleContext:
    full_run_spans: set[str]  # main_pipeline が決定

@dataclass
class FilterResult:
    triggered_tickers: list[str]
    full_run_tickers: list[str]
    skipped_tickers: list[str]
    market_gate_triggered: bool
    trigger_details: dict[str, list[str]]
    status: str  # "FILTER_SKIPPED" / "PARTIAL_TRIGGERED" / "ALL_TRIGGERED"

# status が ALL_TRIGGERED のとき、market_gate_triggered で原因を区別する：
#   market_gate_triggered=True  → 市場全体ゲート発火で全銘柄起動
#   market_gate_triggered=False → 個別判定の結果たまたま全銘柄起動
# Discord 通知にもこの区別を反映し、運用時に理由が追えるようにする。

def evaluate_filter(inputs, config, schedule_context) -> FilterResult
```

### 結合エントリポイント

```python
def run_filter(tickers, pipeline_start, config, schedule_context) -> FilterResult:
    inputs = load_filter_inputs(tickers, pipeline_start)
    return evaluate_filter(inputs, config, schedule_context)
```

## テスト計画

### evaluator 層のユニットテスト（DB モック不要）

**market gate：**
- VIX=35 → 全銘柄トリガー
- VIX delta=5 → 全銘柄トリガー
- VIX delta=2 → トリガーなし
- 10Y yield delta=+20bp → 全銘柄トリガー
- FFR 変化 → 全銘柄トリガー
- FOMC 1日以内 → 全銘柄トリガー
- market gate 発火時に個別判定をスキップしているか

**ticker hard：**
- price_tolerance short 3.5% 超え → トリガー
- latest_quarter 変化 → トリガー（新決算）
- latest_quarter 同一 → トリガーなし
- EPS surprise -12% → トリガー
- 新規 SMA200 割れ (mid) → トリガー
- SMA200: 前回も今回も下 → トリガーなし
- SMA200: 前回下→今回上 → トリガーなし
- SMA200 割れ (short) → トリガーなし

**ticker soft：**
- RSI=25 + close < SMA20 → トリガー
- RSI=25 + close > SMA20 → トリガーなし
- volume 2.5x + close < SMA20 + 当日下落 → トリガー
- volume 2.5x + close < SMA20 + 当日上昇 → トリガーなし

**fail-open：**
- current_tech が None → Monitor 行き
- current_ii が None → Monitor 行き
- prev_ii がない → デルタ無効化、絶対値のみ
- plan 価格がない → Monitor 行き

**full_run：**
- full_run_spans={short, mid}, span=short → バイパス
- full_run_spans={short}, span=mid → フィルター適用
- full_run_spans=空 → 全銘柄フィルター適用

### 統合テスト

1. `python main_pipeline.py --ticker NVDA` → フィルターバイパス確認
2. `python main_pipeline.py --market JP` → フィルター適用確認
3. `python main_pipeline.py --monitor-only` → フィルターバイパス確認
4. 全銘柄スキップ → FILTER_SKIPPED 通知確認
5. 一部トリガー → Monitor が filtered_tickers のみ実行される確認

## 実装ステップ

各ステップは独立してテスト可能。前のステップが完了するまで次に進まない。

### Step 1：evaluator 層（純ロジック） [x]

**目的：** フィルター判定ロジックを DB に一切依存しない形で作り、テストを通す

```
  ↓
Step 2（loader）
  ↓
```
