# 各ブロック不足入力情報

各ブロックに追加すべき入力情報をまとめたドキュメント。

---

## Monitor

### 背景

Monitor の `build_check_prompt()` は、Planning が作成したプラン（`newplan_full`）の一部しか渡していない。
そのため Monitor は「プランの前提が崩れたか」をチェックする際に、損切りラインや保有状況といった重要な情報を参照できていない。

### 現状渡しているもの

| 情報 | 取得元（newplan_full 内） |
|------|------------------------|
| 判定（BUY/HOLD 等） | `verdict`（archive 直下） |
| 投資期間 | `decision.horizon` |
| 確信度 | `decision.confidence` |
| 判定根拠 | `decision.decision_basis` |
| プラン時点の株価 | `data_checks.current_price` |
| テクニカル指標（derived ラベル + raw 数値） | `archive.technical`（別カラム）。実装済み |
| 重要指標 | `archive.important_indicators`（別カラム） |

### 追加すべきもの（全て実装済み）

#### 1. 損切り/利確ライン（実装済み）

| フィールド | 値の例 | 追加する意図 |
|-----------|--------|------------|
| `risk_management.stop_loss_pct` | -8.0 | 現在の株価が損切りラインに近づいているかを判断できる |
| `risk_management.take_profit_pct` | 20.0 | 利確ラインに達したかを判断できる |

Monitor が「株価がプラン時から -7% 下がっている。損切りラインは -8% なので、あと 1% で損切りに達する」と報告できるようになる。

#### 2. 許容価格変動幅（実装済み）

| フィールド | 値の例 | 追加する意図 |
|-----------|--------|------------|
| `data_checks.price_tolerance_pct` | 5.0 | 何%までの変動が「想定内」かの基準を持てる |
| `data_checks.price_block_pct` | 10.0 | この幅を超えたらプラン立て直しの基準を持てる |

現状は Monitor が自分の感覚で「大きく動いた」と判断しているが、プランごとに定義された具体的な閾値を使えるようになる。

#### 3. 投入額/株数（実装済み）

| フィールド | 値の例 | 追加する意図 |
|-----------|--------|------------|
| `portfolio_plan.allocation_jpy` | 150,000 | プランの規模感を把握できる。大きな投入額ほど注意深くチェックすべき |
| `portfolio_plan.quantity` | 5 | 何株の取引かを把握できる |

#### 4. 保有状況（実装済み、Issue #134）

| フィールド | 値の例 | 追加する意図 |
|-----------|--------|------------|
| `holdings.shares` | 15 | 実際に何株持っているかを把握できる |
| `holdings.avg_cost` | 120.50 | 平均取得単価。含み損益を計算できる |
| `holdings.current_price` | 175.20 | 最新の株価（Technical が更新） |

Monitor が「15株保有中（平均$120.50）、現在$175.20 で含み益 +45%」という文脈で前提チェックできるようになる。保有していない銘柄と保有中の銘柄で注意すべきポイントが異なる。

#### 5. プランの鮮度（実装済み）

| フィールド | 値の例 | 追加する意図 |
|-----------|--------|------------|
| `freshness.log_age_days` | 3 | プランが何日前に作られたか。古いプランほど前提が崩れている可能性が高い |
| `freshness.max_allowed_days` | 7 | 投資期間に応じた鮮度上限（短期=2日、中期=7日、長期=30日） |
| `freshness.status` | OK / STALE_REEVALUATE | 鮮度が上限を超えているかどうか |

2日前のプランと2ヶ月前のプランでは注意すべき度合いが変わる。鮮度が上限に近づいている場合は、より慎重にチェックすべき。

#### 6. テクニカル指標の数値（実装済み）

derived ラベル（"neutral" 等）に加えて raw 数値（RSI=41.5、SMA_20=181.44 等）も渡すよう変更済み。
Monitor と Analyzer の両方に適用。

将来的にはどの指標で数値が重要かを調査し、重要な指標のみ数値+閾値、残りはラベルだけにする最適化を行う（Issue #137）。

### 追加後のプロンプトイメージ

```
【銘柄】NVDA
【判定】HOLD
【confidence】LOW
【投資期間】MID
【プラン時点の株価】175.18

【判定根拠】
  - [set1] バリュエーションが割高
  - [set2] 競合の追撃が加速

【プランの鮮度】
  作成からの経過: 3日（上限: 7日）
  鮮度: OK

【リスク管理】
  損切りライン: -8.0%（= $161.17）
  利確ライン: +20.0%（= $210.22）
  許容変動幅: ±5.0%
  プラン停止閾値: ±10.0%

【プランの規模】
  配分額: ¥150,000
  株数: 5

【現在の保有状況】
  保有数: 15株
  平均取得単価: $120.50
  含み損益: +$820.50（+45.4%）

【テクニカル指標】
  トレンド: close_vs_sma20=above, sma20=172.3, close=175.5
  モメンタム: rsi_state=neutral, RSI=69.2
  ボラティリティ: bbands_position=inside_upper_half
  出来高: mfi_state=neutral, MFI=55.0

【重要指標】
  ...
```

### 実装方針

`Monitor/src/main.py` の `build_check_prompt()` に以下を追加する：

1. `newplan_full` から `risk_management` / `data_checks` / `portfolio_plan` / `freshness` の追加フィールドを読む
2. `shared/supabase_client.py` の `get_holding(ticker)` で保有状況を取得する
3. テクニカル指標は derived ラベルに加えて raw 数値も渡す（実装済み）
4. プロンプト文字列に「プランの鮮度」「リスク管理」「プランの規模」「現在の保有状況」セクションを追記する

Monitor は編集不可ブロックのため、実装時にユーザー承認が必要。

## Analyzer

### 背景

Analyzer は銘柄の BUY/SELL/HOLD を議論するが、その銘柄を実際に保有しているかどうかを知らない。
「買うべきか」と「持っているものをどうするか」は全く違う議論であり、保有状況なしでは的確な判断ができない。

### 現状渡しているもの

| 情報 | 取得元 |
|------|--------|
| テクニカル指標（derived ラベル + raw 数値） | `archive.technical` |
| Monitor 結果（NG 理由、変動率） | `archive.monitor` |
| 重要指標（VIX、金利、イベント等） | `archive.important_indicators` |
| Analyzer 設定（レーン数等） | `portfolio_config` |

### 追加すべきもの

#### 1. 保有状況（Issue #134）

| フィールド | 値の例 | 追加する意図 |
|-----------|--------|------------|
| `holdings.shares` | 15 | 「15株持っているが売るべきか」「持っていないが買うべきか」で議論が根本的に変わる |
| `holdings.avg_cost` | 120.50 | 含み益+45%の状態で「まだ持つべきか」「利確すべきか」の議論材料になる |
| `holdings.current_price` | 175.20 | 現在の評価額を把握して議論できる |

現状は Analyst も DA も「この銘柄を持っているかどうか」を知らずに議論している。保有中であれば「含み益が大きいから利確すべき」「まだ上がるから持ち続けるべき」といった議論ができるようになる。

#### 2. 前回のプラン（将来検討）

| フィールド | 値の例 | 追加する意図 |
|-----------|--------|------------|
| `newplan_full.decision.final` | HOLD | 前回どう判断したかを知って「前回の判断はまだ有効か」という議論ができる |
| `newplan_full.decision.decision_basis` | [...] | 前回の判断根拠が変わっていないか確認できる |

現状は毎回ゼロから議論しており、前回の文脈が引き継がれない。優先度は保有状況より低いため将来検討とする。

### 実装方針

Analyzer の初期プロンプト構築部分（`Analyzer/src/main.py`）に保有状況セクションを追加する。

1. `shared/supabase_client.py` の `get_holding(ticker)` で保有状況を取得する
2. Analyst / DA に渡す初期プロンプトに「現在の保有状況」セクションを追記する

Analyzer は編集不可ブロックのため、実装時にユーザー承認が必要。

## Planning

### 背景

Planning はポジションサイジングや配分計算を行うが、既に保有している株を考慮していない。
「ゼロから買う場合の計算」しかできず、「既に持っている分を含めた合計リスク」を評価できない。

### 現状渡しているもの

| 情報 | 取得元 |
|------|--------|
| 最終判定（decision、票数、根拠） | `archive.final_judge` |
| 議論ログ | `archive.lanes` |
| テクニカル指標 | `archive.technical` |
| Monitor 結果 | `archive.monitor` |
| 重要指標 | `archive.important_indicators` |
| 投資パラメータ（予算、リスク上限、損切り幅等） | `portfolio_config` |

### 追加すべきもの

#### 1. 保有状況（実装済み、Issue #134）

| フィールド | 値の例 | 追加する意図 |
|-----------|--------|------------|
| `holdings.shares` | 15 | 既に15株持っている状態で5株追加すると合計20株。合計ポジションのリスクを評価できる |
| `holdings.avg_cost` | 120.50 | 平均取得単価を知ることで、既保有分の含み損益を含めた判断ができる |
| `holdings.current_price` | 175.20 | 既保有分の現在の評価額を計算できる |

実装済み：
- `calc_position_size()` に `existing_investment_jpy` を追加し、既保有分を差し引いた投入上限を算出
- `calc_allocation()` に `existing_investment_jpy` を追加し、残り予算で配分計算
- `PlanSpec` に holdings フィールド追加、YAML 出力に反映
- `plan-generator` のプロンプトに保有状況セクション追加
