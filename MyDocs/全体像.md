# プロジェクト全体像

## 1. プロジェクトの目的

株式投資の「テクニカル指標取得 → 銘柄監視 → AI議論 → プラン生成 → watchlist更新」を完全自動化するシステム。
Claude Code / GLM のマルチエージェント構成で運用する。

---

## 2. パイプラインの2つのルート

| ルート | フロー | 用途 |
|-------|--------|------|
| **定期ルート** | Technical → ImportantIndicators → IndicatorFilter → Monitor → Analyzer → Planning → Watch → ActionLog | 日常の定期監視 |
| **手動ルート** | Admission → Technical → ImportantIndicators → Analyzer → Planning → Watch → ActionLog | 手動で銘柄を追加して分析（Monitor/IndicatorFilterをスキップ） |

### 手動ルートの詳細

- **Admission**: watchlistに未登録なら追加し、archiveレコードを作成（active=True）
- `manual_pipeline.py` の CLI引数: `<ticker> --mode buy|sell --span short|mid|long [--market US|JP] [--display-name 表示名]`
- GitHub Actions `manual-analysis.yml` からも実行可能: `gh workflow run manual-analysis.yml -f ticker=NVDA`

---

## 3. 設計の3大原則（パイプライン総則より）

### 原則1：大ブロック間は直列、ティッカーは並列

- 大ブロック（Technical, Monitor, Analyzer...）は**前ブロックの全銘柄処理が完了してから**次へ進む
- 各大ブロック内では銘柄ごとに `asyncio.gather()` で並列処理
- **例外：ImportantIndicators** は銘柄を逐次処理する。yfinance（Yahoo Finance スクレイピング）が1銘柄あたり4〜5リクエスト必要で、並列化するとIPブロックされるため意図的な設計

### 原則2：DB伝言板方式（疎結合）

- ブロック間のデータは **archive テーブル1レコード**に順番に書き足す
- 環境変数・CLI引数・ファイルによるブロック間の直接データ渡しは**禁止**
- これにより各ブロックを独立して再実行可能（テスト容易性）

### 原則3：ブロック内は小ブロック直列チェーン + レーン並列

- ブロック内部は小ブロックを直列で繋ぐ（直接データ渡し）
- Analyzerのように複数議論が必要な場合は「レーン並列」で対応
- 最終小ブロックのみDBに書き込み（完了の合図）

## 4. 各大ブロックの責務

| ブロック | 何をする | 書き込み先 |
|---------|---------|-----------|
| **Technical** | watchlist全銘柄のテクニカル指標取得 + archiveレコード作成 | `archive.technical` |
| **ImportantIndicators** | VIX・金利・決算・相対強度など市場全体+個別データ取得 | `archive.important_indicators` |
| **IndicatorFilter** | 決定論的にMonitor対象銘柄を絞り込み（LLM不使用、下記4層判定） | オーケストレーター内で判定 |
| **Monitor** | テクニカル+重要指標を参照し OK/NG/ERROR 判定 | `archive.monitor`, `archive.MotivationID`, `archive.active` |
| **Analyzer** | NG銘柄を複数エージェントで並列議論→最終判定 | `archive.lanes`, `archive.final_judge` |
| **Planning** | 最終判定からプランYAMLを生成 | `archive.newplan_full`, `archive.verdict` |
| **Watch** | watchlist更新 + Discord業務通知 + active=False | `watchlist`各カラム, `archive.active` |
| **ActionLog** | action_logテーブルへ記録投入 + holdings同期 | `action_log`, `holdings` |

### 補助ブロック（パイプライン外）

| ブロック | 役割 | 実行タイミング |
|---------|------|--------------|
| **EventScheduler** | FOMC・雇用統計・CPI・決算等の経済イベントを取得し event_master/event_watch テーブルに登録 | annual（年1回 1/2）、monthly（毎月1日）、seed（初期投入） |
| **NightWorker** | archiveレコードをLLMで品質レビュー→GitHub Issue作成（max_reviews=20, dry_run対応）+ words重複統合 | JST 3:05（冗長3回cron） |

## 6. DB伝言板方式の開始条件

各ブロックは前ブロックの完了をDBクエリで検出して動き出す：

| 次ブロック | 開始条件 |
|----------|---------|
| ImportantIndicators | `technical IS NOT NULL AND important_indicators IS NULL` |
| IndicatorFilter | オーケストレーター内でプログラム判定 |
| Monitor | `technical IS NOT NULL AND monitor IS NULL`（IndicatorFilterトリガー銘柄のみ） |
| Analyzer | `active = True AND final_judge IS NULL` |
| Planning | `active = True AND final_judge IS NOT NULL AND newplan_full IS NULL` |
| Watch | `active = True AND status = 'completed' AND newplan_full IS NOT NULL` |
| ActionLog | `active = False AND monitor IS NOT NULL AND archive.id が action_log に未登録` |

## 9. 自動実行の仕組み

### pg_cron + GitHub Actions 二重化トリガー

- **メイン経路**: pg_cron → pg_net → GitHub Actions workflow_dispatch
- **フォールバック**: GitHub Actions 5分ポーリング（event-monitor.yml）
- **重複防止**: 2段階チェック
  1. Supabase DB内（`fire_test_schedules()`）
  2. GitHub Actions上（`get_due_regular_schedules()` + `monitor_last_runs` 150分ウィンドウ）
- **pg_cron死活監視**: `event_watch_check.py`が24h以上成功なしを検知→Supabase Fast Reboot + Discord通知
- **失敗リトライ**: `retry_failed_dispatches` ジョブで自動再試行

## 11. 各大ブロックの実装詳細

### Technical

- `Technical/src/main.py` → 1銘柄処理
  - `process_one_ticker()`: yfinance経由でOHLCV取得 → TechnicalIndicatorFetcherで指標算出
  - `fetch_technical()`: 複数timeframe（1d等）をループ、USD/JPYレートも取得
  - archiveレコード新規作成 or 既存レコード（`technical IS NULL`）を検索して書き込み
  - holdingsスナップショットも同時取得
  - リトライ3回、失敗時Discord通知
- `technical_batch.py` → watchlist active銘柄を`asyncio.gather()`で並列subprocess実行

### ImportantIndicators

- `ImportantIndicators/src/main.py` → オーケストレーター
  - Step1: `fetch_market_data()` で市場全体を1回取得（VIX, 米10年債, FFR, BOJ金利）
  - Step2: 各銘柄ごとに `process_one_ticker()` で個別データ統合
    - `ticker_data.py`: EPS実績vs予想、相対強度（ベンチマーク比較）、出来高異常
    - `event_risk.py`: 直近イベント日数（DB検索）+ Implied Move（米国株のみ、オプション価格から算出）
    - `market_data.py`: yfinance（VIX, ^TNX）+ FRED API（FFR, BOJ金利）
  - 連続失敗検知: 前回archiveと比較し、同一フィールドが連続nullなら1日1回Discord通知

### IndicatorFilter

- `shared/indicator_filter.py` → 2層構成（Loader層 + Evaluator層）
- **Loader層**: DBからデータ収集（現在/前回のarchive, event_master, newplan_full）
- **Evaluator層**: 純ロジック（DBアクセスなし）、以下の4層で判定：

| 判定層 | 条件 | 効果 |
|-------|------|------|
| **定期フル実行（安全ネット）** | short=毎営業日、mid=火金、long=週末 | 該当spanの全銘柄をMonitor |
| **市場全体ゲート** | VIX≥30, VIX delta≥4pt, 10年債delta≥15bp, 金利変更, FOMC/BOJ 1営業日以内（1つでも該当） | 全銘柄をMonitor |
| **個別ハードトリガー** | 価格偏差>tolerance, イベント≤N日, 新規earnings, EPSサプライズ≤-10%, SMA200クロスダウン（1つでも該当） | その銘柄をMonitor |
| **個別ソフトトリガー** | RSI<30 AND close<SMA20、volume_ratio≥2x AND close<SMA20 AND close<prev_close（組み合わせ） | その銘柄をMonitor |

- **fail-open**: データ不足（current_tech/current_ii/plan_priceがnull）の銘柄は安全側に倒してtriggered_tickersに自動追加
- 出力: `FilterResult(triggered_tickers, full_run_tickers, skipped_tickers, market_gate_triggered, trigger_details, status)`

### Monitor

- `Monitor/src/main.py` → 1銘柄処理
  - `run_single()`: 前日テクニカル済みarchive検出 → ImportantIndicators subprocess実行 → `check_one_ticker()`
  - `build_check_prompt()`: plan時株価・判定根拠・リスク管理・現テクニカル・現重要指標・保有状況をプロンプトに組立
  - monitor-checkerエージェント（claude-haiku-4-5 + WebSearch）呼び出し、最大3リトライ
  - 出力: `{result: OK/NG/ERROR, current_price, plan_price, price_change_pct, summary, risk_flags, ng_reason}`
- `Monitor/src/event_watch_check.py` → GitHub Actions cron用
  - イベントwatch検出（event_watchテーブル、30分以内のwatch_at_utc）
  - 定期スケジュール検出（portfolio_config.monitor_schedules）
  - pg_cron死活監視（24h以上成功なし→Supabase Fast Reboot + Discord通知）
  - GITHUB_OUTPUT: `has_watches, markets, skip_spans, event_details`

### Analyzer（最も複雑）

- `Analyzer/src/main.py` → 1銘柄オーケストレーター
  - 市場コンテキスト構築（technical + important_indicators + monitor）
  - 全レーンを`anyio.create_task_group()`で並列起動
- 各レーンのフロー（`lane_orchestrator.py`）:
  1. **議論**（`analyzer_orchestrator.py`）: Analyst↔Devil's Advocateが交互にラウンド制で議論、収束or上限で終了
  2. **Opinion**（`opinion_orchestrator.py`）: 議論ログを読んで独立した意見を並列生成（BUY/SELL/ADD/REDUCE/HOLD）
  3. **Judge**（`judge_orchestrator.py`）: OpinionペアのAGREED/DISAGREED判定
- `final_judge_orchestrator.py`: 全レーン結果を集約
  - 投票ルール: mode=buyなら全会一致のみBUY、mode=reviewなら最多票優先（同票ならHOLD）
  - 出力: `{votes, action_votes, safe_votes, overall_agreement, lane_results, markdown}`

### Planning

- `Planning/src/main.py` → 1銘柄オーケストレーター
  - archive.final_judgeとlanesを取得 → 決定論的計算 → エージェントにcommentary生成委譲 → YAML保存

#### 資金管理の2大原則（plan_calc.py、全て決定論的・LLM不使用）

1. **ポジションサイジング**: 1回の損失を資金の一定%以内に限定する
   - 例：資金100万・許容損失2%（=2万）・損切り5%下 → 投入上限 = 2万÷5% = 40万
   - `calc_position_size()`: 損失許容額からポジション上限を逆算
2. **リスクリワード比（RR比）**: 期待値がプラスのトレードのみ実行
   - `calc_rr_ratio()`: RR比 < min_rr_ratio（デフォルト1.5）なら実行しない

#### その他の計算ステップ

- `check_freshness()`: ログ鮮度チェック（古すぎるログはBLOCK）
- `check_price_deviation()`: 価格ズレ判定（tolerance超でBLOCK）
- `calc_confidence()`: 票差→HIGH/MED/LOW
- `calc_allocation()`: 信頼度→配分%→配分額→株数（ロット丸め）
- `apply_risk_overlay()`: 下記Risk Overlayの制約を乗算

#### Risk Overlay（risk_policy.py、決定論的）

「①制約を先に決める → ②base_sizeを決める → ③制約で絞る」の方針。

**MarketRegime**（VIX×Breadthスコアで判定）:

| 状態 | 条件 | ポジションキャップ | 新規エントリー |
|------|------|------------------|--------------|
| CRISIS | VIX≥35 + Breadth≤-20% | 25% | 禁止 |
| STRESS | VIX≥25 | 60% | 制限付き |
| NORMAL | — | 100% | 可 |
| CALM | VIX<15 | 100% | 可 |

**EventRisk**（イベント接近度で判定）:

| 条件 | 効果 |
|------|------|
| 決算 ≤ 3営業日 | 新規エントリー停止（全面凍結） |
| 決算 4〜7営業日 | ポジションサイズ × 0.7 |
| Tier1マクロイベント（FOMC等） | ポートフォリオ上限 × 0.7 |
| Tier2マクロイベント | ポートフォリオ上限 × 0.85 |

**Portfolio制約**: 保有額率に基づきnew_position_allowedを判定。regime=CRISISではmax_new_positions=0。

- `plan_spec.py`: PlanSpecデータ構造（100+フィールド）+ YAML生成

### Watch

- `Watch/src/main.py` → 1銘柄処理
  - archive.newplan_fullをYAMLパース → watch-summarizerエージェントでサマリー3種生成
    - discussion_summary, new_plan_summary, plan_comparison
  - watchlistテーブル更新（motivation_summary, risk_flags, stock_price等）
  - classify_label()でDiscord通知ラベル判定（URGENT/GOOD_NEWS/WARNING/COMPLETE）
  - archive.active=False に更新（処理完了マーク）

### ActionLog

- `ActionLog/src/pipeline_main.py` → パイプラインバッチ用
  - archive.newplan_full → `auto_populate.populate_action_log()` でaction_log行構築
  - holdings同期: `_sync_holdings_from_logs()` で累積株数・平均取得単価を更新
- `ActionLog/src/calc_engine.py`: cumulative_invested/pnlの連鎖再計算
- `ActionLog/src/main.py`: 月次アクションログ表示WebUI（NiceGUI）

### shared/ 共通基盤

- `supabase_client.py`: archive/watchlist/holdings/action_log/event_master等の全CRUD + `safe_db()`ラッパー
- `llm_client.py`: Claude Code / GLM-4.7 / Codex（GPT-5.4）の3プロバイダー切替、`AgentResult(text, cost, tools_used)` 返却
- `discord_notifier.py`: NotifyPayload → Embed構築 → Webhook送信、7種ラベル対応
- `indicator_filter.py`: 前述のIndicatorFilter実装

## 12. GitHub Actions ワークフロー詳細

### event-monitor.yml（定期パイプラインの中核）
- **トリガー**: 5分間隔cron（:00回避）+ workflow_dispatch（fire_now対応）
- **フロー**:
  1. `event_watch_check.py`でイベントwatch+定期スケジュールをDB検索
  2. 該当なし→即終了 / 該当あり→should_run=true
  3. 各ブロックの依存を`uv sync`でインストール（Technical, Monitor, Analyzer, Planning, ImportantIndicators, Watch）
  4. Codex CLI + auth設定（gpt-5.4）
  5. 市場ごとに`main_pipeline.py`を実行（skip_spans引数対応）
- **環境変数**: CLAUDE_CODE_OAUTH_TOKEN, SUPABASE_*, DISCORD_WEBHOOK_URL, FRED_API_KEY, ZHIPUAI_API_KEY
- **timeout**: 45分

### manual-analysis.yml（手動銘柄分析）
- **トリガー**: workflow_dispatch（ticker必須、mode/span/market/display_name任意）
- **フロー**: `manual_pipeline.py` → Admission → Technical → ImportantIndicators → Analyzer → Planning → Watch → ActionLog
- **Monitor/IndicatorFilterをスキップ**して直接分析
- **timeout**: 60分

### monitor.yml（手動Monitor実行）
- **トリガー**: workflow_dispatch（market/ticker/monitor_only任意）
- **用途**: テスト実行や特定銘柄の手動チェック（定期はevent-monitor.ymlに統合済み）
- `main_pipeline.py`に`--monitor-only`フラグ対応

### event-scheduler.yml（経済イベント登録）
- **トリガー**: 年次（1/2 09:00 JST）+ 月次（毎月1日 09:00 JST）+ workflow_dispatch
- **コマンド**: seed（初期投入）/ annual（年間主要イベント）/ monthly（翌月・翌々月）
- cronトリガー時は日付から自動判定

### night-worker.yml（夜間メンテナンス）
- **トリガー**: JST 3:05（UTC 18:05）+ workflow_dispatch
- **Step1**: archive品質レビュー → GitHub Issue作成（max_reviews=20, dry_run対応）
- **Step2**: words重複統合
- 3回の冗長cron設定（1回目で全件処理済みなら2・3回目は即終了）

### sync-config.yml（設定同期）
- **トリガー**: mainブランチpush時（config/portfolio_config.yml変更検知）+ workflow_dispatch
- `shared/sync_config.py`でYAML→DB同期（dry-run対応）
- **timeout**: 5分（軽量）
