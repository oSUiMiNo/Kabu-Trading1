# 証拠ベース議論改修計画

## 背景

### 解決する問題
1. **fact_id / source_id が常に空欄**（Issue #112）：Discussion の S#/F#/C# 体系が LLM に守られておらず、Planning の decision_basis に根拠情報が入らない
2. **DA が構造的に勝ちすぎる**：全セッションで NOT_BUY_WAIT。BUY 判定がゼロ。議論が機能していない
3. **説得力ある文章に判定が流される**：事実の有無ではなく、文章の説得力で opinion/judge が判断してしまう

### 設計方針
- S#/F#/C#/H#/Q#/R#/A#/D# の8種 ID 体系を廃止し、**証拠表方式**に移行
- ID の通し番号管理を LLM に任せない（LLM が安定して出力できるテーブル形式を採用）
- 「ソースのある事実」と「ソースのない主張」を明確に分離し、後段で証拠の質を定量評価する

---

## 全体像（5ステップ）

```
Step 1: 基盤（プロトコル + analyst/DA）
  ↓
Step 2: 評価レイヤー（opinion + judge）
  ↓
Step 3: 集約（final-judge + orchestrator）
  ↓
Step 4: Planning パイプライン
  ↓
Step 5: 下流消費者（Monitor + NightWorker）
```

### 証拠表の3区分
| 区分 | 内容 | ソース要否 |
|------|------|-----------|
| ソース付き事実 | WebSearch で確認したデータ | 必須（URL + 取得日） |
| 主張 | 事実に基づく推論 | 根拠とした事実を明記 |
| ソースなし主張 | 推測・一般論 | ソースなしと明記 |

### その他の変更
- 本文を**箇条書き必須・散文禁止**に変更（説得力ある文章に流されるのを防止）
- EXPORT yaml を証拠表対応に更新（sourced_facts_count, unsourced_claims_count, evidence_quality）

### 保持した要素
- 追記運用の大原則
- 鮮度ガイドライン
- 数値事実の最小要件（単位・期間・母数）
- 5択スタンス体系（BUY/SELL/ADD/REDUCE/HOLD）

### DA 勝ちすぎ問題への対策（Step 1 分）
- Analyst にリスクセクションを書かせない（DA の役割と明示）
- 両方に主張数の上限5件を設定（数の不均衡を解消）

## DA 勝ちすぎ問題への対策（Step 1〜2 で実装）

ログ分析で発見した5つの原因と対策：

| 原因 | 対策 | 実装先 |
|------|------|--------|
| Analyst が自分のリスクを書く | Analyst にリスクセクションを書かせない（DA の役割） | Step 1: analyst.md |
| DA の方が数字で具体的に見える | evidence_score で証拠の質を定量比較 | Step 2: judge.md |
| ネガティブ主張は常に正しく見える | opinion に「リスクの実現蓋然性も評価せよ」と指示 | Step 2: opinion.md |
| DA のテキスト量が多い | 両方に主張数の上限を設ける | Step 1: analyst.md, devils-advocate.md |
| LLM の保守バイアス | opinion に「証拠の質に明確な差があれば証拠優先、同程度なら安全側」の優先順位を明示 | Step 2: opinion.md |

## 制約

### パイプライン総則（`MyDocs/パイプライン総則.md`）
- **大ブロック間の情報伝達は DB（archive テーブル）経由のみ**。環境変数・コマンド引数・ファイル等による大ブロック間の直接データ受け渡しは禁止
- **大ブロック内の小ブロック間は直接渡し**（手段は問わない）。今回の証拠表関連の状態管理は Discussion ブロック内で完結するため、この制約内で実現可能
- **各大ブロックは前ブロックの全銘柄完了を待ってから開始**。Discussion の証拠表形式変更が Planning に影響する場合、archive テーブルの final_judge カラム経由で伝達する
- **archive テーブルへの書き込み責務**：Discussion → lanes, final_judge / Planning → newplan_full, verdict。今回の変更でこの責務分担は変えない

### AgentResult の制約
- `call_agent()` は最終テキスト・コスト・ツール名のみ返す
- WebSearch の URL やクエリはオーケストレーター側で取得不可
- → URL は LLM のテキスト出力に依存。Opus 本番環境では捏造リスクは低い

### 編集権限
- Discussion / Planning / Monitor / EventScheduler は原則編集不可。本改修はユーザー承認済み
- Discussion が記録したログ（archive の lanes, final_judge 等）は絶対に変更しない
- watchlist テーブルへの書き込みは Watch ブロックのみ

### 鮮度ガイドラインの強化（Step 1 で実装済み）
- 「常に最優先ではない」の但し書きを削除
- 鮮度不明の情報は「ソースなし主張」に自動分類
