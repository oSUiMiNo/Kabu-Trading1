# プロジェクト理解

株式投資の自動監視・分析・プラン策定システム。
定期的に保有/監視銘柄の状態をチェックし、プランの前提が崩れた銘柄を検出したら、AI エージェント群による再分析とプラン再策定を自動実行する。

---

## 1. システムの目的

ユーザーが策定した投資プラン（例：「NVDAを○○ドルで買い、××ドルで利確、△△ドルで損切り」）に対し、以下を自動化する。

1. **定期監視**：プランの前提条件（株価、ニュース、市場環境）が維持されているか定期チェック
2. **異常検出**：前提が崩れた（NG）銘柄を検出
3. **再分析**：NG銘柄を複数AIエージェントで議論し、BUY/SELL/HOLD等の最終判定を出す
4. **プラン再策定**：最終判定に基づき、ポジションサイジング・リスクリワード比を含む新プランを自動生成
5. **通知**：結果をDiscordに通知

## 4. パイプラインの構造

本パイプラインは **大ブロック** を直列に並べたもの。
オーケストレーター（`main_pipeline.py`）が大ブロックを決められた順番で実行する。

### 4.1 マクロ構造（大ブロック間）

#### 実行順序：直列（前ブロック全完了を待つ）

```
Technical（全銘柄完了）
  → Monitor（全銘柄完了）
    → Analyzer（全銘柄完了）
      → Planning（全銘柄完了）
        → Watch（全銘柄完了）
```

- オーケストレーターは大ブロックを順番に呼び出すだけ。ティッカー並列化の責務は持たない

#### ティッカー処理：並列

各大ブロックは対象銘柄を並列処理する能力を自ら持つ（`asyncio.gather` 相当）。
あるティッカーの処理が他のティッカーの完了を待つことはない。

#### 情報伝達：DB（伝言板方式）

大ブロック間のデータは **DB の archive テーブルを介して** 受け渡す。
ブロック同士は直接データをやり取りしない。環境変数・コマンド引数・ファイル等による大ブロック間の直接的なデータ受け渡しは禁止。

| 書き込み元ブロック | 書き込み先カラム | 読み取り元ブロック |
|-----------------|----------------|-----------------|
| Technical | `technical` | Monitor |
| Monitor | `monitor`, `MotivationID`, `motivation_full`, `active`, `status` | Analyzer（NG 判定の参照） |
| Analyzer | `lanes`, `final_judge` | Planning |
| Planning | `newplan_full`, `verdict` | Watch |
| Watch | watchlist テーブル更新 + `active`=False | Monitor（次回サイクル） |

全ブロックが同一の archive レコードに順番に書き足す。ブロックごとに別レコードを作らない。

### 4.2 ミクロ構造（大ブロック内部）

#### 小ブロックの直列チェーン

大ブロックの内部は **小ブロック** で構成される。
1つの銘柄に対する処理は、小ブロックを先頭から末尾まで順番に実行する。

例（Monitor ブロック内、1銘柄の処理）：

```
get_latest_archivelog_with_newplan(ticker)  # DB から最新プランを取得
  → build_check_prompt()                 # プロンプト生成
    → call_agent()                       # エージェント呼び出し
      → parse_monitor_result()           # 結果パース
        → update_archivelog()            # DB に書き込み（ブロック完了の証跡）
```

#### 情報伝達：直接渡し（DB を介さない）

ブロック内の小ブロック間では、前の小ブロックの結果を直接次の小ブロックへ渡す。
渡し方はメモリ上のデータをそのまま渡す場合もあれば、ファイルに出力してそのパスを伝える方法もある。
手段は問わないが、DB（第三者）を介さず直接届けるという点が重要。

> **例外：** ブロック内の最終小ブロックだけは DB に書き込む。これが「このブロックの処理が完了した」という次ブロックへの合図になる。

#### レーン並列

大ブロック内部では、小ブロックの直列チェーン（＝**レーン**）が複数並列に走る場合がある。

例（Analyzer ブロック内）：

```
Lane A: opinion → judge → final_judge  ┐
Lane B: opinion → judge → final_judge  ├─ 互いに待たず並列実行
Lane C: opinion → judge → final_judge  ┘
```

#### バリエーション

| パターン | 構成 |
|---------|------|
| 単一ブロック | パイプラインを形成しない単体の処理 |
| 単純直列 | 1本のレーンのみ |
| レーン並列 | 複数レーンが並列実行 |
| 混在 | 直列区間とレーン並列区間が混在 |

### 4.3 マクロ・ミクロ対比

| 観点 | マクロ（大ブロック間） | ミクロ（大ブロック内部） |
|------|----------------------|----------------------|
| 接続 | 大ブロックが直列 | 小ブロックが直列（レーン） |
| 並列化 | 各大ブロックがティッカー並列を自律実行 | レーン単位で並列化 |
| 情報伝達 | archive テーブル経由（伝言板方式） | 小ブロック間の直接受け渡し（手段は問わない） |
| オーケストレーターの役割 | 大ブロックを順番に実行するだけ | 関与しない |

### 4.4 構造図

```
【大ブロック間】

  Technical ブロック       Monitor ブロック          Analyzer ブロック         Planning ブロック
  ┌──────────────┐       ┌───────────────┐         ┌──────────────────┐       ┌─────────────────┐
  │ NVDA ─────→ DB │      │ NVDA ──────→ DB │        │ NVDA ──────→ DB  │       │ NVDA ──────→ DB │
  │ AAPL ─────→ DB │─DB─→│ AAPL ──────→ DB │──DB──→│ AAPL ──────→ DB  │──DB──→│ AAPL ──────→ DB │
  │ ...  ─────→ DB │      │ ...  ──────→ DB │        │ ...  ──────→ DB  │       │ ...  ──────→ DB │
  └──────────────┘       └───────────────┘         └──────────────────┘       └─────────────────┘
   全完了を待って次へ       全完了を待って次へ          全完了を待って次へ

【大ブロック内（1銘柄の処理）】

  小ブロック1 → 小ブロック2 → 小ブロック3 → DB書込
  （直列・直接渡し、他の銘柄を待たない）
```

### 4.5 設計の意図

| 観点 | 方針 | 理由 |
|------|------|------|
| ブロック間を直列にする | 前ブロック全完了を待つ | Analyzer は Monitor の全結果を見てから始めるべき |
| ブロック内を並列にする | 銘柄ごとに同時実行 | 銘柄間に依存がないため待つ理由がない |
| ブロック間伝達を DB にする | archive テーブルを介す | ブロックを独立して再実行できる疎結合を保つ |
| ブロック内伝達を直接渡しにする | 手段は問わず直接次の小ブロックへ | 同一責務の処理を DB（第三者）で分断する必要がない |

### 4.6 同一archiveレコード方式

全ブロックが1銘柄につき1つのarchiveレコードに順番に書き足す。

```
archive ID: 1つのレコード
  Technical  → archive.technical を書き込み
  Monitor    → archive.monitor, MotivationID, active を書き込み
  Analyzer   → archive.lanes, final_judge を書き込み
  Planning   → archive.newplan_full, verdict を書き込み
  Watch      → watchlist更新 + archive.active = False
```

### 4.7 次ブロックの開始条件

| ブロック | 開始条件 |
|---------|---------|
| Monitor | `technical IS NOT NULL AND monitor IS NULL` |
| Analyzer | `active = True AND final_judge IS NULL` |
| Planning | `active = True AND final_judge IS NOT NULL AND newplan_full IS NULL` |
| Watch | `active = True AND status = 'completed' AND newplan_full IS NOT NULL` |

OK判定の銘柄は `active=False` となるため、Analyzer以降には流れない。NGのみが下流に進む。

## 6. 各ブロックの役割

### Technical

watchlist全銘柄のテクニカル指標（SMA、RSI、MACD、ボリンジャーバンド等22種以上）をyfinance経由で取得し、archiveレコードを新規作成して`archive.technical`に書き込む。パイプラインの起点であり、全銘柄のarchiveレコードを作る唯一のブロック。

自作ライブラリ [TechnicalIndicatorFetcher](https://github.com/oSUiMiNo/TechnicalIndicatorFetcher.git) を使用。

### Monitor

各銘柄の既存プランの前提が崩れていないかをチェックし、OK/NG/ERRORを判定する。

- `monitor-checker.md`（GPT-5.4 / Codex SDK + WebSearch）がチェックを担当
- 株価・ニュースを調査し、プランの前提条件と照合
- NG → `active=True`（Analyzer以降へ進む）
- OK → `active=False, status=completed`
- ERROR → 最大3回リトライ後、Discord通知のみ（下流に進まない）
- リスクフラグ：OKでも注目すべき変化があれば付与（例：`価格乖離超過（下落）`、`決算未達`）

### Analyzer（編集不可）

NG銘柄を複数AIエージェントで議論し、最終判定（BUY/SELL/HOLD等）を出す。

- 複数レーンで並列議論（Analyst ↔ Devil's Advocate → Opinion → Judge → Final Judge）
- 証拠表方式：ソース付き事実とソースなし主張を分離し、証拠の質を定量評価
- `archive.lanes`（議論ログ）と`archive.final_judge`（最終判定）を書き込み

**LLMバックエンド**：Claude Code と Z.AI GLM の2つに対応。`.env.local` で切り替え。

| 項目 | 内容 |
|------|------|
| 切り替え | `ANALYZER_LLM_PROVIDER=glm` / `ANALYZER_GLM_MODEL=glm-4.7` |
| GLMプラン | Coding Plan Lite（年間） |
| エンドポイント | `https://api.z.ai/api/coding/paas/v4/` |
| 利用可能モデル | `glm-4.7`、`glm-4.7-flash`（無料） |
| クォータ | 5時間あたり80〜120プロンプト / 週間上限約400 |

### Planning（編集不可）

final_judgeに基づき、具体的な投資プランYAMLを生成する。

- **ポジションサイジング**：許容損失額から投入上限を逆算
- **リスクリワード比**：利確/損切りの比率を評価（1未満なら実行不可）
- **配分計算**：confidence × budget で投入額を決定（ポジションサイジングの制限あり）
- `plan_calc.py`（固定ルール計算）+ `plan-generator.md`（commentary生成）で構成
- `archive.newplan_full`にプランYAMLを書き込み

### Watch

パイプラインの最終ブロック。プラン結果をwatchlistに反映し、業務通知を送る。

- `archive`のデータからサマリーを生成し、`watchlist`テーブルを更新
- `classify_label()`で通知ラベルを判定：【緊急】【警告】【朗報】
- `archive.active = False`に更新（パイプライン完了の証）

### ImportantIndicators

市場全体＋個別銘柄の定量データを API で取得し、`archive.important_indicators` に書き込む。
パイプラインに自律的に並ぶのではなく、Monitor（毎回）や Analyzer（手動起動時、今日のデータがなければ）が起動時に呼び出す方式。

6グループの指標：
1. **EventRisk**：イベントまでの日数、Implied Move
2. **MarketRegime**：VIX
3. **金利動向**：米10年債利回り、FRB政策金利、日銀政策金利
4. **Earnings & Guidance**：EPS実績 vs 予想
5. **Relative Strength**：ベンチマーク指数・セクター対比
6. **Volume Anomaly**：5日平均出来高比、ドル出来高

### EventScheduler（編集不可）

決算・経済イベント（FOMC、CPI等）のスケジュールをDBに登録・管理する。
MonitorやImportantIndicatorsがイベント接近度を計算する際の情報源。

### NightWorker

archive の品質レビューを行い、問題があればGitHub Issueを作成する。旧名 ArchiveReview。
用語集統合機能（`words_consolidator.py`）も含む。

### ActionLog

銘柄ごとの月次アクションログを表示するローカル Web アプリ（NiceGUI）。

- 1画面 = 1銘柄 × 1か月のプラン履歴を表示
- カラム：TS、アクションログ、ストーリー、入れたお金、総資産、損益
- archive.newplan_full および monitor 結果から自動投入（Watch 完了時）
- ユーザーが値を修正すると関連項目が自動再計算
- 月の先頭に引き継ぎ文（GLM-4.7 で自動生成）
- ストーリー列は月の流れに沿った文章（GLM-4.7 で生成）
- 起動：`cd ActionLog/src && uv run python main.py` → http://localhost:8080

## 7. 定期実行の仕組み

### 二重化構成

| 経路 | 精度 | 役割 |
|------|------|------|
| **pg_cron**（メイン） | 分単位で正確 | 狙った時刻ぴったりに workflow_dispatch で GitHub Actions を起動 |
| **GitHub Actions cron**（フォールバック） | 5〜78分のばらつき | pg_cron 死亡時の保険。5分間隔ポーリング |

### 実行フロー

```
pg_cron / GH Actions cron
  → event_watch_check.py
    → DB のスケジュール定義と照合（許容幅: -3分〜+90分）
    → 重複チェック（monitor_last_runs で150分以内なら skip）
    → マッチ → main_pipeline.py 起動
```

### 登録済みスケジュール（pg_cron）

| ジョブ | 発火時刻（UTC） | 対象（JST） |
|--------|---------------|------------|
| monitor_JP_AM | 月〜金 01:10 | 日本株 10:10 |
| monitor_JP_PM | 月〜金 07:00 | 日本株 16:00 |
| monitor_US_AM | 日〜木 15:00 | 米国株 0:00 |
| monitor_US_PM | 日〜木 21:30 | 米国株 6:30 |
| monitor_WEEKEND | 土日 01:00 | 全銘柄 10:00 |

### GitHub Actions ワークフロー

| ワークフロー | スケジュール | 内容 |
|-------------|------------|------|
| `event-monitor.yml` | 5分間隔 24時間稼働 | イベント watch + 定期スケジュール検出 → パイプライン起動 |
| `monitor.yml` | 手動（workflow_dispatch） | テスト・特定銘柄の手動チェック |
| `event-scheduler.yml` | 年1回・月1回 | FOMC・雇用統計等の経済イベント取得・DB登録 |
| `night-worker.yml` | JST 3:05 / 4:05 / 5:05 | archive 品質レビュー → GitHub Issue 作成 |
| `sync-config.yml` | push 時自動 | `config/portfolio_config.yml` → DB 反映 |

## 8. DB構造

### 主要テーブル

| テーブル | 用途 |
|---------|------|
| `archive` | パイプライン出力の蓄積。1銘柄1レコードに全ブロックが書き足す |
| `watchlist` | 監視対象銘柄と最新サマリー |
| `portfolio_config` | 全設定値（YAML → DB同期） |
| `monitor_last_runs` | 重複実行防止（スケジュールラベルごとの最終実行時刻） |
| `event_master` / `event_watch` | イベント管理（EventScheduler管轄） |
| `archive_reviews` | archive 品質レビュー結果（NightWorker管轄） |
| `holdings` | 保有銘柄情報（株数・平均取得単価・現在価格） |
| `action_log` | ActionLog アプリ用のアクション履歴 |
| `action_log_handoff` | ActionLog の月別引き継ぎ文キャッシュ |

### archiveテーブルの主要カラム

| カラム | 書き込み元 | 型 |
|--------|-----------|-----|
| `technical` | Technical | JSONB |
| `important_indicators` | ImportantIndicators | JSONB |
| `monitor` | Monitor | JSONB |
| `MotivationID` | Monitor | INTEGER |
| `active` | Monitor / Watch | BOOLEAN |
| `status` | Monitor / Planning | TEXT |
| `lanes` | Analyzer | JSONB |
| `final_judge` | Analyzer | JSONB |
| `newplan_full` | Planning | JSONB |
| `verdict` | Planning | TEXT |

### 書き込み責務の絶対ルール

- 各ブロックは自分の担当カラムのみ書き込み、他ブロックが書いたカラムは読み取り専用
- Analyzerが記録したログ（lanes, final_judge等）は絶対に変更しない
- watchlistテーブルへの書き込みはWatchブロックのみ

## 11. Analyzerの議論構造

### 証拠表方式

LLMの議論で「事実」と「主張」を明確に分離する。

| 区分 | 内容 | ソース要否 |
|------|------|-----------|
| ソース付き事実 | WebSearchで確認したデータ | 必須（URL + 取得日） |
| 主張 | 事実に基づく推論 | 根拠とした事実を明記 |
| ソースなし主張 | 推測・一般論 | ソースなしと明記 |

### 議論フロー（1レーン）

```
Analyst（買い推奨派）
  → Devil's Advocate（反対意見）
    → Opinion（評価・証拠品質比較）
      → Judge（evidence_score算出・判定）
        → Final Judge（全レーン統合・最終判定）
```

- 複数レーンを並列実行し、Final Judgeが全レーンを統合
- 箇条書き必須・散文禁止（文章の説得力に流されることを防止）

## 12. 資金管理ロジック（Planningブロック）

### ポジションサイジング

「1回の損失を資金の○%以内に抑える」から投入上限を逆算。

```
投入上限 = 許容損失額 / 損切り幅(%)
例：資金300万 × 許容5% = 15万円 / 8% = 187.5万円
```

既存のconfidenceベース配分と比較し、小さい方を採用。

### リスクリワード比（RR比）

```
RR比 = 利確までの幅 / 損切りまでの幅
例：利確+20% / 損切り-8% = 2.5
```

RR比 < 1.0 のトレードは `quantity=0` で実行不可。

## 13. 設計原則

| 原則 | 内容 |
|------|------|
| **伝言板方式** | ブロック間のデータ受け渡しはDB（archive）経由のみ。直接渡し禁止 |
| **疎結合** | 各ブロックは独立して再実行可能。DBにレコードがあれば単体で動く |
| **関心の分離** | フォルダごとにモジュール分離。他ブロックの改変禁止（閲覧OK） |
| **編集不可ブロック** | CLAUDE.md の編集可・不可リストに従う（ユーザー承認で例外あり） |
| **並列化はレーン単位** | ブロック内の並列化は直列チェーン（レーン）を並列に回す |
| **APIで定量データ取得** | 正確な数値が必要な場面はAPI。WebSearchはLLMの定性判断用途のみ |
