# 重要指標ブロック 要件定義書

---

## 1. 概要

### 目的

市場全体の環境と個別銘柄の定量データを API で自動取得し、archive テーブルに記録する大ブロックを新設する。
Monitor・Analyzer・Planning が archive からこのデータを読み取ることで、LLM の自発的な調査に頼らず安定して重要指標をカバーできるようになる。

設計方針の詳細は「入力指標と組み込み方針.md」を参照。

### 利用データソース

| データソース | 対象 | 認証 | 制限 |
|-------------|------|------|------|
| **yfinance** | 株価、出来高、VIX、利回り、EPS、オプション等 | 不要 | 目安：1時間360リクエスト程度（非公式） |
| **FRED API** | FRB政策金利、日銀金利等のマクロ経済指標 | 無料APIキー | 1分120リクエスト。日次・月次制限なし |

---

## 2. パイプライン上の位置づけ

### 呼び出し方式

重要指標ブロックは独立した大ブロックだが、パイプラインに自律的に並ぶのではなく、Monitor または Analyzer が起動時に呼び出す。

```
Monitor 起動時：
  Monitor → Technical を呼ぶ（毎回）
         → 重要指標 を呼ぶ（毎回）
         → Monitor 本体の判定処理

Analyzer 手動起動時：
  Analyzer → 鮮度チェック（今日のデータがあるか）
            → なければ Technical + 重要指標 を呼ぶ
            → Analyzer 本体の分析処理
```

| 呼び出し元 | 重要指標の扱い |
|-----------|--------------|
| Monitor | 毎回実行（常に最新データで判定） |
| Analyzer（手動） | 今日のデータがあればスキップ、なければ実行 |

### 書き込み先

`archive.important_indicators`（JSONB カラム、新設）

Technical ブロックが作成済みの archive レコードに書き足す。重要指標ブロックが archive レコードを新規作成することはない。

## 4. 機能仕様

### 4.1 データ取得の2層構造

重要指標は「市場全体データ」と「個別銘柄データ」に分かれる。

| 層 | 取得タイミング | 対象 |
|----|--------------|------|
| **市場全体データ** | パイプライン実行ごとに1回 | VIX、金利（全銘柄共通） |
| **個別銘柄データ** | 銘柄ごとに1回 | EPS、出来高、Relative Strength、Implied Move（米国株のみ） |

市場全体データは1回取得して全銘柄の archive レコードに同じ値を書き込む。

### 4.2 指標グループ別の取得仕様

#### グループ1：EventRisk

| 指標 | 取得方法 | 備考 |
|------|---------|------|
| イベントまでの日数 | EventScheduler の DB（`event_master`/`event_watch` テーブル）から当該銘柄・市場の直近イベントを検索し、今日からの日数を計算 | FOMC、CPI、雇用統計、決算発表 |
| Implied Move（米国株のみ） | yfinance `ticker.option_chain(nearest_expiry)` で ATM 付近の Call+Put の mid price を合算し、現在株価に対する % を算出 | 最も近い満期のオプションを使用。日本株は初回実装では省略（Issue #130） |

**Implied Move 計算ロジック（米国株）：**
```
1. ticker.options で満期日一覧を取得
2. 最も近い満期を選択
3. ticker.option_chain(date) で Call/Put チェーンを取得
4. 現在株価に最も近い strike（ATM）を特定
5. ATM Call の mid = (bid + ask) / 2
6. ATM Put の mid = (bid + ask) / 2
7. implied_move_pct = (call_mid + put_mid) / current_price * 100
```

#### グループ2：MarketRegime

| 指標 | 取得方法 | 備考 |
|------|---------|------|
| VIX | yfinance `yf.download('^VIX', period='5d')` の直近終値 | 市場全体データ（1回取得）。RSP/SPY 比率（Advance-Decline 代替）は初回実装では省略（Issue #130） |

#### グループ3：金利動向

| 指標 | 取得方法 | 備考 |
|------|---------|------|
| 米10年債利回り | yfinance `yf.download('^TNX', period='5d')` の直近終値 | % 表記（例：4.35） |
| FRB政策金利 | FRED API `fred.get_series('DFF')` の直近値 | 日次実効レート |
| 日銀政策金利 | FRED API で日本の政策金利シリーズを取得 | 更新は政策決定会合後のみ |

#### グループ4：Earnings & Guidance

| 指標 | 取得方法 | 備考 |
|------|---------|------|
| EPS 実績 vs 予想 | yfinance `ticker.get_earnings_history()` から直近四半期の `epsActual`, `epsEstimate`, `surprisePercent` を取得 | 直近4四半期分 |
| 売上実績 vs 予想 | yfinance `ticker.revenue_estimate` から取得 | 取得できない場合は省略。アナリスト予想推移（ガイダンス修正の代替）は初回実装では省略（Issue #130） |

#### グループ5：Relative Strength

| 指標 | 取得方法 | 備考 |
|------|---------|------|
| vs ベンチマーク指数 | yfinance で銘柄と指数（米国株：`^GSPC`、日本株：`^N225`）の過去3ヶ月リターンを取得し、差分を計算 | `rs_vs_index = ticker_return - index_return` |
| vs 同一セクター | yfinance `ticker.info['sector']` でセクターを特定し、対応するセクターETF（XLK, XLF 等）との過去3ヶ月リターン差分を計算 | 日本株はセクターETFが限定的なため省略可 |

**セクター → ETF マッピング（米国株）：**

| セクター | ETF |
|---------|-----|
| Technology | XLK |
| Healthcare | XLV |
| Financials | XLF |
| Consumer Discretionary | XLY |
| Consumer Staples | XLP |
| Energy | XLE |
| Industrials | XLI |
| Materials | XLB |
| Utilities | XLU |
| Real Estate | XLRE |
| Communication Services | XLC |

#### グループ6：Volume Anomaly

| 指標 | 取得方法 | 備考 |
|------|---------|------|
| 5日平均出来高比 | yfinance `ticker.history(period='1mo')` から直近出来高 / 過去5日平均出来高 を計算 | 1.0 = 平均的、2.0 = 2倍 |
| ドル出来高 | 直近の `Close × Volume` | 流動性の代理指標としても Planning が参照 |

### 4.3 実行モード

| モード | コマンド例 | 用途 |
|--------|-----------|------|
| 全銘柄一括 | `python main.py` | Monitor からの呼び出し |
| 単一銘柄 | `python main.py --ticker AAPL` | Analyzer からの呼び出し |
| archive 指定 | `python main.py --ticker AAPL --archive-id 123` | 特定の archive レコードに書き込み |

全銘柄一括モードでは、市場全体データを1回取得した後、各銘柄の個別データを並列取得する。

## 6. レート制限対策

### yfinance 対策（必須）

| 対策 | 実装方法 | 目的 |
|------|---------|------|
| **curl_cffi セッション** | `curl_cffi.requests.Session(impersonate="chrome")` を yfinance に渡す | TLS フィンガープリンティングによるブロック回避 |
| **ランダム遅延** | 各銘柄のデータ取得間に `random.uniform(1, 3)` 秒の待機 | 機械的パターンの回避 |
| **バッチ取得** | 市場全体データは `yf.download(['^VIX', '^TNX'], period='5d')` で一括取得 | リクエスト数の削減 |
| **指数バックオフリトライ** | 429 エラー時は 2秒→4秒→8秒→16秒 で再試行、5回失敗で諦め | 一時的な制限への対応 |

### FRED API 対策

1分120リクエストの制限に対してこのプロジェクトは数リクエスト/回のため、特別な対策は不要。APIキーを `.env.local` で管理する。

## 7. エラーハンドリング

### 基本方針

一部の指標取得に失敗してもブロック全体を止めない。取得できたデータだけを archive に書き込む。後段ブロック（Monitor/Analyzer/Planning）は `null` チェックして利用する。

### エラー種別と対処

| エラー種別 | 原因 | 対処 |
|-----------|------|------|
| yfinance データ取得失敗 | ネットワーク障害、Yahoo 側の制限 | 指数バックオフリトライ（最大5回）→ 失敗した指標は `null` で記録 |
| yfinance IP ブロック | 過剰アクセス、TLS 検出 | curl_cffi セッションで回避。それでもブロックされた場合は `null` で記録 |
| FRED API 失敗 | ネットワーク障害、キー失効 | リトライ → 失敗時は `null` で記録 |
| オプションデータなし | 日本株、小型株 | Implied Move フィールドを `null` で記録。日本株の ATR 代替は初回実装では省略（Issue #130） |
| EPS/Revenue データなし | IPO 直後、未上場転換 | 該当フィールドを `null` で記録 |
| セクター情報なし | ETF、指数、一部外国株 | Relative Strength のセクター比較を省略 |

### Discord 通知：連続失敗検知 + 1日1回制限

2つの条件を組み合わせて通知を制御する：
1. **連続失敗**：同じエラー種別が2回連続で発生した場合のみ通知（一時エラーは無視）
2. **1日1回制限**：同じエラー種別の通知は1日に1回だけ（yfinance 仕様変更等で全銘柄に同じエラーが出ても通知は1通）

**判定ロジック：**

```
今回の取得完了後：
  null のフィールドがある
    → 前回も同じフィールドが null か確認（連続失敗チェック）
      → 前回は正常 → 通知なし（一時エラー）
      → 前回も null → 今日このエラー種別で既に通知済みか確認（1日1回チェック）
        → 通知済み → 抑制
        → 未通知 → Discord 通知を送信し、通知済みとして記録
```

**エラー種別の分類**：通知の抑制は「エラー種別」単位で行う。銘柄ごとではない。

| エラー種別 | 例 |
|-----------|-----|
| `market.vix` 取得失敗 | VIX が全銘柄で null → 1通 |
| `earnings.eps` 取得失敗 | 5銘柄で EPS が null → 1通（「EPS 取得失敗（対象: NVDA, AAPL, ...）」） |
| `event_risk.implied_move` 取得失敗 | 米国株3銘柄で失敗 → 1通 |

**通知対象と非対象：**

| 種別 | 通知 | 理由 |
|------|------|------|
| 市場全体データ（VIX、金利等）が2回連続 null | **通知する**（1日1回） | 全銘柄に影響する重大な障害 |
| 個別銘柄の指標（EPS、RS等）が2回連続 null | **通知する**（エラー種別で1日1回） | yfinance の仕様変更を示唆。該当銘柄を一覧で通知 |
| 日本株の Implied Move が null | 通知しない | 初回実装ではオプションデータのない銘柄は省略（Issue #130） |
| 日本株のセクター比較が省略 | 通知しない | 仕様上の正常動作 |
| 売上データ（revenue）が null | 通知しない | yfinance で安定取得が難しく、null が頻発しうる |

**通知メッセージの例：**

```
【警告】重要指標の連続取得失敗
エラー種別：earnings.eps 取得失敗
該当銘柄：NVDA, AAPL, TSLA
連続失敗回数：2回
最終成功：2026-03-21
→ yfinance の EPS データ取得に持続的な問題が発生している可能性があります
```

```
【エラー】市場全体データの連続取得失敗
エラー種別：market.vix 取得失敗
連続失敗回数：3回
最終成功：2026-03-20
→ yfinance の市場データ取得に持続的な問題が発生している可能性があります
```

## 9. 既存ブロックへの影響

### Monitor

| 項目 | 現在 | 変更後 |
|------|------|--------|
| 重要指標の取得 | なし | 起動時に重要指標ブロックを呼び出し、archive に書き込んでから判定処理を開始 |
| archive 参照 | `archive.technical`, `archive.monitor` | 上記に加えて `archive.important_indicators` を参照可能に |

### Analyzer

| 項目 | 現在 | 変更後 |
|------|------|--------|
| 重要指標の取得 | なし | 手動起動時に鮮度チェック → 今日のデータがなければ重要指標ブロックを呼び出し |
| archive 参照 | `archive.technical`, `archive.monitor` | 上記に加えて `archive.important_indicators` を参照可能に |

### Planning

| 項目 | 現在 | 変更後 |
|------|------|--------|
| 流動性情報 | なし | `archive.important_indicators.volume.dollar_volume` をポジションサイズ調整の参考にできる |
