# パイプライン総則

## 1. 全体像

本パイプラインは **大ブロック** を直列に並べたものである。
オーケストレーターが大ブロックを決められた順番で実行することで、一連の処理が完了する。

```
[大ブロックA] → [大ブロックB] → [大ブロックC] → ...
```

### 大ブロックの一覧

| 大ブロック | 役割の一言説明 |
|-----------|-------------|
| **Monitor** | watchlist 銘柄の現状をチェックし OK/NG/ERROR を判定 |
| **Discussion** | NG 銘柄を複数エージェントで議論し最終判定を出す |
| **Planning** | 最終判定からプラン YAML を生成し DB に保存 |
| **Watch** | プラン結果をもとに watchlist のサマリーを更新／追加 |
| **EventScheduler** | 決算・カレンダーイベントや定期監視スケジュールを管理 |

---

## 2. マクロ構造（大ブロック間）

### 2.1 実行順序：直列（前ブロック全完了を待つ）

大ブロックは必ず前のブロックの **全銘柄処理が完了してから** 次のブロックを開始する。

```
Monitor（全銘柄完了）
  → Discussion（全銘柄完了）
    → Planning（全銘柄完了）
      → Watch（全銘柄完了）
```

### 2.2 ティッカー処理：並列

各大ブロックは、対象銘柄を **並列**処理する能力を自ら持つ。
あるティッカーの処理が他のティッカーの完了を待つことはない。

- オーケストレーターは大ブロックを順番に呼び出すだけであり、ティッカー並列化の責務を持たない
- オーケストレーターの起動主体は人間・定期発火デーモンなど

### 2.3 情報伝達：DB（伝言板方式）

大ブロック間のデータは **DB の `archive` テーブルを介して**受け渡す。
ブロック同士は直接データをやり取りしない。

| 書き込み元ブロック | 書き込み先テーブル・カラム | 読み取り元ブロック |
|-----------------|------------------------|-----------------|
| Monitor | `archive`：`id`, `created_at`, `ticker`, `mode`, `span`, `status`, `monitor`, `MotivationID`, `motivation_full`, `active`<br>`watchlist`：`MotivationID` | Discussion（NG 判定の参照） |
| Discussion | `archive`：`lanes`, `final_judge` | Planning |
| Planning | `archive`：`newplan_full`, `verdict` | Watch |
| Watch | `watchlist`：`motivation_summary`, `discussion_result`, `discussion_summary`, `new_plan_summary`, `risk_flags`, `plan_comparison`, `stock_price`<br>`archive`：`active`（False に戻す） | Monitor（次回サイクルの参照） |

次ブロックの開始条件は `archive.active = True` + ステージ固有条件で決定する。

| ブロック | 開始条件（DB クエリ） |
|---------|---------------------|
| Discussion | `active = True AND final_judge IS NULL` |
| Planning | `active = True AND final_judge IS NOT NULL AND newplan_full IS NULL` |
| Watch | `active = True AND status = 'completed' AND newplan_full IS NOT NULL` |

`MotivationID` は一度セットしたら変更しない（恒久的な動機分類）。`active` はパイプライン処理中のみ True。

## 3. ミクロ構造（大ブロック内部）

### 3.1 基本構成：小ブロックの直列チェーン

大ブロックの内部は **小ブロック** で構成される。
1つの銘柄に対する処理は、小ブロックを先頭から末尾まで順番に実行する。
他の銘柄の完了を待つ必要はなく、自分の処理が終わり次第、次の小ブロックへ進む。

例（Monitor ブロック内、1銘柄の処理）：

```
get_latest_archivelog_with_newplan(ticker)  # DB から最新プランを取得
  → build_check_prompt()                 # プロンプト生成
    → call_agent()                       # エージェント呼び出し
      → parse_monitor_result()           # 結果パース
        → update_archivelog()            # DB に書き込み（ブロック完了の証跡）
```

### 3.2 情報伝達：直接渡し（DB を介さない）

ブロック内の小ブロック間では、前の小ブロックの結果を直接次の小ブロックへ渡す。
渡し方はメモリ上のデータをそのまま渡す場合もあれば、ファイルに出力してそのパスを伝える間接的な方法もある。手段は問わないが、DB（第三者）を介さず直接届けるという点が重要である。

> **例外：** ブロック内の最終小ブロックだけは DB に書き込む。これが「このブロックの処理が完了した」という次ブロックへの合図になる。

### 3.3 レーン並列

大ブロック内部では、小ブロックの直列チェーン（＝**レーン**）が複数並列に走る場合がある。
各レーンは他のレーンの完了を待たずに、自分の処理を最後まで走りきる。

例（Discussion ブロック内）：

```
Lane A: opinion → judge → final_judge  ┐
Lane B: opinion → judge → final_judge  ├─ 互いに待たず並列実行
Lane C: opinion → judge → final_judge  ┘
```

### 3.4 並列化の方針

ブロック内でティッカー並列以外の並列処理が必要になった場合は、**レーン単位で並列化する**。

- 正しい形：直列チェーン（レーン）を並列に回す
- 誤った形：ブロック単位で並列にし、それを直列に繋ぐ

### 3.5 バリエーション

大ブロック内部の構成は、以下のいずれにもなりうる。

| パターン | 構成 |
|---------|------|
| 単一ブロック | パイプラインを形成しない単体の処理。並列実行される場合でも、1ブロック＝1レーンとみなす |
| 単純直列 | 1本のレーンのみ |
| レーン並列 | 複数レーンが並列実行 |
| 混在 | 直列区間とレーン並列区間が混在 |

## 4. まとめ

### 構造図

```
【大ブロック間】

  Monitor ブロック          Discussion ブロック         Planning ブロック
  ┌───────────────┐         ┌──────────────────┐       ┌─────────────────┐
  │ NVDA ──────→ DB │        │ NVDA ──────→ DB  │       │ NVDA ──────→ DB │
  │ AAPL ──────→ DB │──DB──→│ AAPL ──────→ DB  │──DB──→│ AAPL ──────→ DB │
  │ ...  ──────→ DB │        │ ...  ──────→ DB  │       │ ...  ──────→ DB │
  └───────────────┘         └──────────────────┘       └─────────────────┘
   全完了を待って次へ          全完了を待って次へ

【大ブロック内（1銘柄の処理）】

  小ブロック1 → 小ブロック2 → 小ブロック3 → DB書込
  （直列・直接渡し、他の銘柄を待たない）
```

### マクロ・ミクロ対比

| 観点 | マクロ（大ブロック間） | ミクロ（大ブロック内部） |
|------|----------------------|----------------------|
| 接続 | 大ブロックが直列 | 小ブロックが直列（レーン） |
| 並列化 | 各大ブロックがティッカー並列を自律実行 | レーン単位で並列化 |
| 情報伝達 | archive テーブル経由（伝言板方式） | 小ブロック間の直接受け渡し（手段は問わない） |
| オーケストレーターの役割 | 大ブロックを順番に実行するだけ | 関与しない |

### 設計の意図

| 観点 | 方針 | 理由 |
|------|------|------|
| ブロック間を直列にする | 前ブロック全完了を待つ | Discussion は Monitor の全結果を見てから始めるべき |
| ブロック内を並列にする | 銘柄ごとに同時実行 | 銘柄間に依存がないため待つ理由がない |
| ブロック間伝達を DB にする | archive テーブルを介す | ブロックを独立して再実行できる疎結合を保つ |
| ブロック内伝達を直接渡しにする | 手段は問わず直接次の小ブロックへ | 同一責務の処理を DB（第三者）で分断する必要がない |
