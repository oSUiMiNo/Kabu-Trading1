# 入力指標と組み込み方針

市場や銘柄から取得する外部情報（入力データ）の定義と、プロジェクトへの組み込み方針。
Claude・ChatGPT・Codex（OpenAI o3）の3者議論を経て策定。

入力データと計算ロジック（出力指標）の違いについては「Planningロジック（資金管理ルール）.md」を参照。


## 入力データ6グループ（重要指標ブロック → archive → 後段ブロックが読み取り）

「市場全体 → 個別銘柄」をカバーする6グループ。
1グループに複数フィールドを含めてよいが、Analyzerが区別できるよう内訳を明記する。

ブロック間のデータ受け渡しは **DB（archiveテーブル）経由の伝言板方式**。
重要指標ブロックが取得したデータをarchiveに書き込み、Monitor/Analyzer/Planningがそこから読み取る。

| # | グループ | 具体的な指標 | 読み取り先 | 何がわかるか |
|---|---------|------------|-----------|------------|
| 1 | **EventRisk** | FOMC・CPI・雇用統計・決算発表までの日数、Implied Move（ATMストラドル価格から算出する期待変動幅） | Analyzer / Planning | 「大きく動きやすい予定」が近いか、どの程度荒れそうか |
| 2 | **MarketRegime** | CBOE VIX（市場全体の恐怖度を示す指数）、NYSE Advance-Decline（値上がり銘柄数 − 値下がり銘柄数） | Analyzer / Planning | 市場全体の恐怖度と、上げ下げに参加している銘柄の広さ |
| 3 | **金利動向** | 米10年債利回り（US 10-Year Treasury Yield）、FRB政策金利（Federal Funds Rate）、日銀政策金利 | Analyzer / Planning | 金融政策の方向性。株式市場全体の大局観に直結 |
| 4 | **Earnings & Guidance** | EPS実績 vs アナリスト予想（1株あたり利益のサプライズ）、売上実績 vs 予想、ガイダンス修正（会社が出す業績見通しの上方/下方修正） | Analyzer | 過去の決算結果だけでなく、今後の見通しがどう変化しているか |
| 5 | **Relative Strength** | 銘柄パフォーマンス vs ベンチマーク指数（S&P 500、TOPIX等）、銘柄 vs 同一セクター | Analyzer | その銘柄が市場全体や同業より強いか弱いか |
| 6 | **Volume Anomaly** | 5日平均出来高比（直近5日の平均に対する当日出来高の倍率）、ドル出来高（株価×出来高で算出する売買代金） | Analyzer | 出来高の急増＝何かが起きているシグナル |

補助入力：**Liquidity**（ビッド・アスクスプレッド、板厚、日次ドル出来高）→ Planning
流動性が低い銘柄ではポジションサイズを自動で落とす判断材料として使う。

> **未組み込みの指標**：Issue #14 で挙げられた信用買残・信用売残・信用倍率は現在どのグループにも含まれていない。組み込み先の検討が必要。

### API取得可否

| 指標 | API取得 | 備考 |
|------|:---:|------|
| イベントまでの日数 | 可 | EventSchedulerのDBから日数計算 |
| Implied Move | 難 | オプション価格データが必要。米国株は有料APIで可能、日本株はほぼ不可 |
| CBOE VIX | 可 | Yahoo Finance等 |
| NYSE Advance-Decline | 可 | 市場データAPI |
| 米10年債利回り | 可 | FRED API等 |
| FRB政策金利 | 可 | FRED API |
| 日銀政策金利 | 可 | 公開データ |
| EPS実績 vs 予想 | 可 | Financial Modeling Prep等 |
| 売上実績 vs 予想 | 可 | 同上 |
| ガイダンス修正 | 難 | 構造化データとして提供するAPIが少ない。IR資料やニュースからの読み取りが必要 |
| 銘柄 vs ベンチマーク指数 | 可 | 株価データから比率を計算 |
| 銘柄 vs 同一セクター | 可 | 同上 |
| 5日平均出来高比 | 可 | 出来高データから計算 |
| ドル出来高 | 可 | 株価×出来高で計算 |
| ビッド・アスクスプレッド | 難 | Level 2（板情報）データが必要、有料API |
| 板厚 | 難 | 同上 |
| 信用買残 | 可 | 東証（日本株）/ FINRA Short Interest（米国株） |
| 信用売残 | 可 | 同上 |
| 信用倍率 | 可 | 買残÷売残で計算 |

API取得可能：15個 / 難：4個（Implied Move、ガイダンス修正、スプレッド、板厚）

> **補足**：Liquidityの「日次ドル出来高」とVolume Anomalyの「ドル出来高」は同じ計算（株価×出来高）。取得は1回で両方に使える。

### データソースと取得方式

使用するデータソースは **yfinance**（Python ライブラリ）と **FRED API** の2つのみ。どちらも無料。

| データソース | 対象 | 認証 | 制限 |
|-------------|------|------|------|
| **yfinance** | 株価、出来高、VIX、利回り、EPS、オプション等ほぼ全般 | 不要 | 公式制限なし（目安：1時間360リクエスト程度） |
| **FRED API** | FRB政策金利、日銀金利等のマクロ経済指標 | 無料APIキー | 1分120リクエスト。日次・月次制限なし |

yfinance は Yahoo Finance の非公式ライブラリであり、Yahoo 側の仕様変更やアクセス制限の影響を受ける可能性がある。FRED API は米連邦準備銀行の公式APIで安定性が高い。

#### 「難」指標の代替方式

元々「有料APIが必要」とされた4指標のうち、2つは yfinance で取得可能。

| 指標 | 代替方式 | 取得元 |
|------|---------|--------|
| Implied Move（米国株） | yfinance の `option_chain()` でオプション価格を取得し、ATM Call+Put から計算 | yfinance |
| Implied Move（日本株） | ATR（Average True Range）で代替 | yfinance |
| ガイダンス修正 | yfinance の `eps_trend`（アナリスト予想推移：7日前/30日前/90日前）で代替 | yfinance |
| スプレッド・板厚 | 取得不可。ドル出来高を流動性の代理指標として使用 | ― |

#### yfinance のレート制限対策

yfinance は2024年11月以降、Yahoo 側の制限が厳格化されている。このプロジェクトの規模（watchlist 20銘柄程度、1回50〜100リクエスト）なら問題ないが、以下の対策をコードに組み込む。

| 対策 | 内容 | 目的 |
|------|------|------|
| **ランダム遅延** | リクエスト間に1〜3秒のランダムな待機を挿入 | 機械的なアクセスパターンの回避 |
| **curl_cffi セッション** | 通信の指紋を Chrome ブラウザに偽装 | TLSフィンガープリンティングによるブロック回避 |
| **指数バックオフリトライ** | 失敗時に待機時間を倍々に増やしながら再試行（2秒→4秒→8秒→16秒→諦め） | Yahoo 側の一時的な制限への対応 |

#### WebSearch ではなく API を使う理由

定量データ（VIX、利回り、株価等）は API で取得する。WebSearch は使わない。

| 観点 | API | WebSearch |
|------|-----|-----------|
| 数値の正確性 | 構造化データそのまま（100%） | LLM が検索結果を解釈するため誤差あり |
| データ鮮度 | タイムスタンプ付きで明確 | いつの値か検証不能 |
| 再現性 | 同じクエリ→同じ結果 | 同じ質問でも42〜79%の確率で異なる値が返る |
| コスト | API直叩きで低コスト | LLM呼び出し＋Web検索で高コスト |

最新モデル（Claude Opus 4.6等）でも金融タスクでの幻覚率は完全にはゼロにならないため、正確な数値が必要な場面では API が唯一の確実な手段。WebSearch は Monitor/Analyzer の定性判断（ニュース調査）に留める。

### 選定の根拠

| 層 | カバー範囲 | 該当グループ |
|----|-----------|------------|
| 市場全体の環境 | 危険度・参加度・金融政策 | EventRisk, MarketRegime, 金利動向 |
| 個別銘柄の評価 | 業績変化・相対的な強さ・需給の異常 | Earnings & Guidance, Relative Strength, Volume Anomaly |

### 重要な設計判断とその理由

| 論点 | 結論 | 理由 |
|------|------|------|
| VIX + Implied Moveの統合 | **統合NG** | VIXは市場全体のセンチメント、Implied Moveは個別銘柄のイベントボラ。レイヤーが異なり、同一スケールに押し込むと解釈が壊れる |
| 出来高 + 流動性の統合 | **統合NG** | 出来高急増は「何かが起きている」シグナル、流動性は「執行コスト」で用途が逆。統合すると誤解釈リスクがある |
| Breadth vs 相対強度 | **相対強度を独立枠に** | BreadthはVIXと合わせて市場レジーム内で管理。相対強度は銘柄選定に直結するためAnalyzer入力として独立枠にする |
| 金利動向の独立性 | **独立枠を維持** | VIX/Breadth（センチメント）と金利（金融政策方向性）は性質が異なり、同居させると情報が薄まる |

### Implied Moveが取れない場合の代替

オプション未上場銘柄（日本株の中小型等）ではImplied Moveが取得できない。
その場合は以下で代替する（要検証）。

- 過去の決算発表前後のヒストリカルボラティリティ
- ATR（Average True Range）


## 組み込み方針：「指示」ではなく「入力データ」で渡す

### 原則

Analyzerに「この指標を見ろ」と指示すると視野が狭まる。
現状のAnalyzerは自由に調査することで、銘柄固有の重要ファクター（地政学リスク、個別事業の進捗等）を自発的に拾えている。

重要指標は **Analyzerへの入力データとして添える** ことで、視野を狭めずにカバー率を上げる。

### データ取得の呼び出し方式

Technicalブロックと重要指標ブロックは独立した大ブロックだが、自律的にパイプラインに並ぶのではなく、MonitorまたはAnalyzerが起動時に呼び出す。

```
Monitor起動時：
  Monitor → Technical を呼ぶ（毎回）
         → 重要指標 を呼ぶ（毎回）
         → Monitor本体の判定処理

Analyzer手動起動時：
  Analyzer → 鮮度チェック（今日のデータがあるか）
            → なければ Technical + 重要指標 を呼ぶ
            → Analyzer本体の分析処理
```

| 呼び出し元 | Technical・重要指標の扱い |
|-----------|------------------------|
| Monitor | 毎回実行（常に最新データで判定） |
| Analyzer（手動） | 今日のデータがあればスキップ、なければ実行 |

### ブロック別の役割

| ブロック | やること | やらないこと |
|---------|---------|-------------|
| **EventScheduler** | 経済イベント・決算発表の予定と重要度をDBに登録する | 銘柄判断 |
| **Technical** | テクニカル指標を取得し、archiveに書き込む | 指標の解釈・判断 |
| **重要指標** | 6グループのデータを取得し、archiveに書き込む | 指標の解釈・判断 |
| **Monitor** | Technical・重要指標を呼び出した後、archiveのデータを参照し、既存プランの前提が崩れていないか判定する | ― |
| **Analyzer** | 鮮度チェック後、必要ならTechnical・重要指標を呼び出し、archiveのデータを含め自由な視野で銘柄を分析 | 特定指標を見ろという縛りは受けない |
| **Planning** | archiveからデータを読み取り、ポジションサイジング・リスクリワード比を必ず計算。ドル出来高に応じてサイズ調整 | 銘柄の良し悪しの判断 |

### archive に格納するデータの例

重要指標ブロックが取得した指標データは archive に書き込む。
Monitor/Analyzer/Planning は archive からこのデータを読み取って利用する。

```
市場環境：
- EventRisk: 次回FOMC まで 3日 / 6702 決算発表まで 12日
- MarketRegime: VIX 25.3（やや高め）/ Breadth 42%（参加率低め）
- 金利: 米10年債 4.35% / 日銀政策金利 0.50%

銘柄データ（6702）：
- Earnings & Guidance: 前回EPS予想¥85→実績¥102（+20%サプライズ）/ ガイダンス上方修正+8%
- Relative Strength: TOPIX電機セクター比で強め（+3.2%）
- Volume Anomaly: 5日平均比 +180%（急増）
- Liquidity: スプレッド 0.08% / 板厚 十分
```


## Archive分析結果（2026-03-15時点）

直近のarchive（id=110: U、id=111: 6702）を分析した結果。
入力データとしての仕組み導入前の現状。

| グループ | 現状のカバー状況 | 詳細 |
|---------|:---:|------|
| EventRisk | なし | イベント接近度への言及なし |
| MarketRegime | なし | VIX・Breadthともに言及なし |
| 金利動向 | たまに出る | U(id=110)で「10Y米債4.35%」に言及あり。6702では一切なし |
| Earnings & Guidance | 一部 | 将来の決算への言及はあるが、予想vs実績の定量分析はなし |
| Relative Strength | なし | 指数・同業比較の言及なし |
| Volume Anomaly | なし | 出来高への言及ゼロ |

結論：6グループ中、仕組みとして確実にカバーされているものは0個。
LLMが自発的に触れることはあるが、安定して考慮されるにはデータ入力が必要。


## 策定経緯

### 3者議論の概要

| 参加者 | 主な主張 |
|--------|---------|
| **Claude** | VIX・出来高・金利・決算サプライズ・ポジションサイジング・リスクリワードの6指標。「市場→個別→資金管理」の3層カバーを重視 |
| **ChatGPT** | イベント接近度・Breadth・相対強度・ガイダンス修正・流動性・期待変動の6指標。EventScheduler活用と「指標vs計算ロジック」の分離を重視 |
| **Codex（o3）** | Claude案を主軸としつつGPT案の入力指標を取り込み、「入力データ6 + Planningロジック2」に分離するのがベストと結論 |

### 各案の評価

- **Claude案が正しかった点**：Planningに資金管理を含める必要性、金利の独立性、出来高の重要性
- **GPT案が正しかった点**：イベント接近度のEventScheduler活用、相対強度の有用性、ポジションサイジング/RRは「指標ではなくロジック」という分類
- **統合の結果**：両者の良い部分を取り込み、構造的な整理（入力とロジックの分離）を加えた
