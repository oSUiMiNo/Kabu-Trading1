# ActionLog テスト方針

## 1. 単体テスト（1つのファイルだけを対象にテスト）

### calc_engine.py

| テストケース | 内容 |
|-------------|------|
| 連鎖再計算（正常系） | 3行の money_in から cumulative_invested と pnl を正しく計算 |
| money_in 編集後の連鎖 | 中間行の money_in を変更 → 後続行の cumulative_invested と pnl が連動更新 |
| total_assets 編集 | 1行の total_assets を変更 → その行の pnl のみ変化、他行は不変 |
| 空データ | 行がない場合に例外が出ない |
| 売り（負の money_in） | SELL/REDUCE で money_in が負値になり、cumulative_invested が減少 |

### auto_populate.py

| テストケース | 内容 |
|-------------|------|
| BUY の変換 | decision=BUY → action_text="xxx円買った", money_in=allocation_jpy |
| HOLD の変換 | decision=HOLD → action_text="様子見", money_in=0 |
| NO_BUY の変換 | decision=NO_BUY → action_text="保留", money_in=0 |
| 二重投入防止 | 同じ archive_id で2回呼んでもエラーにならない（DB が自動で弾く） |

---

## 2. 結合テスト（DB と組み合わせてテスト）

### data_service.py（DB の読み書き）

- action_log の追加・取得・更新が正常動作
- action_log_handoff の追加・取得が正常動作
- 同じ銘柄×archive の組み合わせを2回入れようとしたら DB が防いでくれる
- 銘柄と日付で月別に絞り込むクエリが正しく動く

---

## 3. UI 手動テスト

| # | 操作 | 期待結果 |
|---|------|---------|
| 1 | アプリ起動 | ブラウザに銘柄選択画面が表示 |
| 2 | 銘柄を選択 | 当月のアクションログテーブルが表示 |
| 3 | 月ナビ `→` をクリック | 翌月に切り替わる |
| 4 | 前月がある月に移動 | 引き継ぎ文が月頭に表示 |
| 5 | `入れたお金` セルを編集 | 後続行の累計と損益が自動更新 |
| 6 | `総資産` セルを編集 | その行の損益のみ自動更新 |
| 7 | ストーリーの長い行 | 行高がストーリー文量に応じて伸びる |

---

## 4. 一気通貫テスト（Watch → DB → 画面まで全部通して動かす）

1. Watch を1銘柄で実行（`python watch_batch.py --ticker NVDA`）
2. action_log テーブルにレコードが投入されたことを Supabase で確認
3. ActionLog アプリを起動して該当銘柄・月を表示
4. 自動投入されたデータが正しく表示されることを確認
