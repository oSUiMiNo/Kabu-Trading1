# ActionLog 方針

## フレームワーク選定：NiceGUI

### 選定理由

| 要件 | NiceGUI の対応 |
|------|---------------|
| セル単位の編集 + 自動再計算 | 表計算コンポーネント（AG Grid）を内蔵しており、セルの値が変わったとき自動で Python の処理が動く |
| 可変行高（ストーリー文長） | テキスト量に応じて行の高さが自動で伸び縮みする機能あり |
| 月ナビゲーション | ボタンクリックで Python 関数を直接呼べる |
| Python エコシステム | 純 Python で完結（HTML/JS 不要） |
| メンテナンス性 | Python だけで全て書ける。非エンジニアでもコードが追いやすい |
| 依存の軽さ | インストールが簡単で、他のブロックに影響を与えない |
| ローカル専用 | デフォルトでローカル実行 |

### 不採用の候補

| 候補 | 不採用理由 |
|------|-----------|
| Streamlit | セルを1つ変えるだけで画面全体が再描画される。行の高さ調整も非対応 |
| FastAPI + HTML/JS | 画面を HTML/JS で一から書く必要があり、メンテナンスが大変 |
| Reflex | 裏で Node.js が動くため、セットアップが煩雑 |

## データ保存方針：Supabase 新テーブル

- `action_log` テーブルと `action_log_handoff` テーブルを Supabase に新規作成
- 別 PC からもデータにアクセス可能
- 既存の `shared/supabase_client.py` に読み書き用の関数を追加

---

## データ投入方針

### 自動投入のタイミング：Watch 完了時

- Watch の `notify(payload)` 呼び出し後に投入関数を呼ぶ
- この時点で `payload.beginner_summary` が取得可能
- Watch への変更は 2-3 行の追加のみ（import + 関数呼び出し）

### NO_BUY / NOT_BUY_WAIT

- 「保留」としてアクションログに記録する
- `money_in = 0`（投入額なし）

### ユーザー編集の保護

- ユーザーが手動で直したフィールド名を DB に記録しておく（`user_overrides` カラム）
- パイプラインが再実行されても、ユーザーが直した値は自動値で上書きされない

---

## 既存 archive データの扱い

既存の archive レコード（`newplan_full` があるもの）も action_log に自動投入する。

- アプリ初回起動時、または手動操作で、過去の archive から一括取り込みを行う
- `beginner_summary` は過去分には保存されていないため、story 列は空欄になる（ユーザーが後から手動入力可）
- 投入後はユーザーが内容を修正できる（実際の行動と異なる場合など）
