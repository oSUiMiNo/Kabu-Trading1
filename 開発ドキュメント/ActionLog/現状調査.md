# ActionLog 現状調査

## ActionLog が読み取る既存データ

### archive テーブルの newplan_full カラム

Planning ブロックが生成したプラン YAML が `archive.newplan_full` に保存されている。
ActionLog はここからデータを読み取って自動投入する。

以下は newplan_full の中で ActionLog が使うフィールド：

| フィールド（newplan_full 内のパス） | 内容 |
|----------------------------------|------|
| `decision.final` | 判定（BUY/SELL/ADD/REDUCE/HOLD/NO_BUY/NOT_BUY_WAIT） |
| `portfolio_plan.allocation_jpy` | その銘柄への配分額（円）。例：NVDAに150,000円 |
| `portfolio_plan.quantity` | 株数（ロット調整済み） |
| `data_checks.current_price` | プラン作成時の株価 |
| `decision.confidence` | 確信度（HIGH/MED/LOW） |
| `decision.horizon` | 投資期間（SHORT/MID/LONG） |

### beginner_summary（ストーリーの元データ）

- **生成元**：`shared/discord_notifier.py` の `generate_beginner_summary()`
- **生成者**：`Monitor/.claude/commands/notify-summarizer.md`（claude-haiku-4-5）
- **フォーマット**：3〜5文、200文字以内、冒頭にアクション、初心者向け平易な日本語
- **現状の問題**：Discord 送信後に破棄され、どこにも保存されていない

### 生成タイミングと取得方法

`Watch/src/main.py:181` の `await notify(payload)` 呼び出し後、`payload.beginner_summary` に値がセットされる。
ただしこの値は Watch の処理が終わると消えるため、**消える前に action_log テーブル（新設）の `story` カラムに保存する**。

```
Watch/src/main.py の処理フロー（101-190行）：
1. archive 取得
2. watch-summarizer でサマリー生成
3. watchlist 更新
4. notify(payload) ← この中で beginner_summary 生成、Discord に送信
5. ★ ここで payload.beginner_summary を action_log テーブルに保存 ★
   （Discord に送った直後、まだメモリ上に残っているこのタイミングで拾う）
6. archive.active = False ← この後 Watch の処理が終わり、payload は消える
```

## ActionLog が従う既存の実装パターン

### DB アクセス

- DB の読み書き関数は `shared/supabase_client.py` に集約する（プロジェクト全体の慣習）
- 全ての DB 呼び出しは `safe_db()` ラッパーで囲んでエラーを吸収する
- 各ブロックは `sys.path.insert` で shared フォルダを参照する

### ディレクトリ構成

```
<ブロック名>/
├── src/
│   ├── main.py
│   ├── pyproject.toml
│   └── .venv/
└── .claude/
    └── commands/
```

- 各ブロック独立した venv（uv で管理）
- Python >= 3.10
